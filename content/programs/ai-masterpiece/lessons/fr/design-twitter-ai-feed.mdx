# Concevoir le flux IA de Twitter

Il s'agit d'une **étude de cas finale** — vous combinerez tout, de AI Sketch à AI Polish, pour concevoir un flux de médias sociaux complet alimenté par l'IA.

---

## L'énoncé du problème

> Concevez un système qui sert des tweets personnalisés à plus de 500 millions d'utilisateurs en temps réel, en classant le contenu par pertinence, récence et potentiel d'engagement.

**Ce qui rend cela difficile :**

- **Échelle :** 500 millions d'utilisateurs actifs quotidiens, plus de 500 millions de tweets/jour
- **Latence :** Le flux doit se charger en moins de 200 ms
- **Fraîcheur :** Les nouveaux tweets apparaissent en quelques secondes
- **Personnalisation :** Chaque utilisateur voit un flux unique
- **Démarrage à froid :** Nouveaux utilisateurs sans historique

<Callout type="info">
C'est exactement le type de question posée lors des entretiens de niveau senior+ chez Meta, Google, Twitter/X et LinkedIn. Cela teste simultanément les connaissances en DSA, conception système ET ML.
</Callout>

## Phase 1 : Modélisation des données (Fondamentaux DSA)

### Entités principales

```
User { id, name, followers[], following[], interests[] }
Tweet { id, authorId, content, media[], timestamp, metrics }
Engagement { userId, tweetId, type: view|like|retweet|reply, timestamp }
```

### Structures de données clés

| Structure | Objectif | Pourquoi celle-ci ? |
|-----------|----------|---------------------|
| **Hash Map** | Recherche utilisateur → profil | Accès O(1) par userId |
| **Sorted Set** | Classement du fil d'actualité | Insertion O(log n) + requêtes par plage |
| **Bloom Filter** | Déduplication « déjà vu » | Appartenance à un ensemble efficace en mémoire |
| **Inverted Index** | Recherche de contenu | Associe les tokens → IDs de tweets |
| **Graph (adjacency list)** | Connexions sociales | Relations abonnement/abonné |

<ThinkAboutIt>
Pourquoi utiliser un Bloom Filter au lieu d'un HashSet classique pour la détection « déjà vu » ? Avec 500 millions d'utilisateurs voyant chacun plus de 100 tweets/jour, un HashSet consommerait des téraoctets. Un Bloom Filter utilise environ 1 % de cette mémoire avec un taux de faux positifs acceptable (environ 1 %).
</ThinkAboutIt>

## Phase 2 : Architecture système

### Composants de haut niveau

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│  Tweet       │────▶│  Fan-out     │────▶│  Timeline Cache  │
│  Ingestion   │     │  Service     │     │  (Redis Sorted   │
└─────────────┘     └──────────────┘     │   Sets)          │
                                          └────────┬────────┘
                                                   │
┌─────────────┐     ┌──────────────┐              ▼
│  User        │────▶│  Ranking     │     ┌─────────────────┐
│  Request     │     │  Service     │────▶│  Feed Response   │
└─────────────┘     └──────────────┘     └─────────────────┘
                           │
                    ┌──────┴──────┐
                    │  ML Model   │
                    │  Service    │
                    └─────────────┘
```

### Stratégie de Fan-Out : Approche hybride

**Fan-out on write** pour les utilisateurs avec moins de 10 000 abonnés :
- Quand un utilisateur tweete, pousser vers tous les fils d'abonnés
- Lectures à faible latence (le fil est pré-calculé)
- O(abonnés) écritures par tweet

**Fan-out on read** pour les célébrités (plus de 10 000 abonnés) :
- Ne pas pousser — fusionner au moment de la lecture
- Évite d'écrire dans des millions de fils
- Latence de lecture légèrement plus élevée

```
function handleNewTweet(tweet, author):
    if author.followerCount < CELEBRITY_THRESHOLD:
        // Fan-out on write
        for follower in author.followers:
            timelineCache.add(follower.id, tweet, score=tweet.timestamp)
    else:
        // Fan-out on read — store in celebrity tweets pool
        celebrityTweets.add(author.id, tweet)
```

<Callout type="warning">
L'approche hybride est celle que Twitter utilise réellement. Le fan-out-on-write pur ne passe pas à l'échelle pour les comptes avec des millions d'abonnés (un tweet = des millions d'écritures en cache).
</Callout>

## Phase 3 : Modèle de classement ML

### Ingénierie des features

Le modèle de classement prédit : **P(l'utilisateur interagit avec le tweet)**

| Catégorie de features | Exemples |
|----------------------|----------|
| **Features utilisateur** | intérêts, schémas d'engagement passés, heures d'activité |
| **Features tweet** | âge, type de média, longueur, hashtags, popularité de l'auteur |
| **Features croisées** | historique d'interaction utilisateur-auteur, chevauchement thématique |
| **Features contextuelles** | heure de la journée, type d'appareil, vitesse de connexion |

### Classement en deux étapes

**Étape 1 — Génération de candidats (rapide, large) :**
- Récupérer environ 1 000 candidats du cache de fil + pool de célébrités
- Utiliser un modèle léger (régression logistique) pour le scoring initial
- Budget de latence : 50 ms

**Étape 2 — Classement fin (lent, précis) :**
- Re-classer les 200 meilleurs candidats
- Réseau neuronal profond avec mécanisme d'attention
- Considérer les features croisées et les patterns séquentiels
- Budget de latence : 100 ms

```
function rankFeed(userId, candidates):
    // Stage 1: Coarse ranking
    scored = candidates.map(tweet =>
        ({tweet, score: lightweightModel.predict(userId, tweet)})
    )
    top200 = scored.sortBy(s => -s.score).slice(0, 200)

    // Stage 2: Fine ranking
    reranked = deepModel.batchPredict(userId, top200)

    // Stage 3: Business rules (diversity, freshness boost)
    return applyBusinessRules(reranked)
```

### Diversité et fraîcheur

Le modèle de classement seul ne montrerait que du contenu viral. Appliquer des règles post-classement :

- **Diversité thématique :** Pas plus de 3 tweets consécutifs sur le même sujet
- **Diversité d'auteur :** Répartir les tweets du même auteur
- **Boost de fraîcheur :** Décroissance temporelle logarithmique — les tweets récents obtiennent un score bonus
- **Exploration :** 5 % des emplacements du flux montrent du contenu en dehors de la bulle de l'utilisateur

## Phase 4 : Gestion des cas limites

### Problème du démarrage à froid

Les nouveaux utilisateurs n'ont pas d'historique d'engagement. Solutions :

1. **Intérêts d'intégration :** Demander aux utilisateurs de choisir 3 à 5 sujets lors de l'inscription
2. **Repli sur le contenu populaire :** Montrer les tweets tendance dans les sujets sélectionnés
3. **Explorer-exploiter :** Explorer agressivement pendant les 48 premières heures, puis exploiter
4. **Priors démographiques :** Utiliser la localisation, la langue, l'appareil pour les signaux initiaux

### Mises à jour en temps réel

```
// WebSocket connection for live updates
connection.onTweet(tweet => {
    if (isHighPriority(tweet)):  // Breaking news, close friend
        injectToFeed(tweet, position=TOP)
    else:
        showNewTweetsButton(count++)
})
```

### Abus et sécurité

- **Modération de contenu :** Un classifieur ML signale le contenu nuisible avant le classement
- **Limitation de débit :** Token bucket par utilisateur (voir AI Craft — Rate Limiter)
- **Détection de spam :** Analyse basée sur les graphes des patterns de suivi + similarité de contenu

Lors d'un vrai entretien, attendez-vous à des questions de suivi :
