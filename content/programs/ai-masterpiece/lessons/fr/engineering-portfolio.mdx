# Portfolio d'ingénieur

Un portfolio solide fait la différence entre *« qui est cette personne ? »* et *« accélérons le processus pour ce candidat. »* Cette leçon vous apprend à en construire un qui se démarque.

---

## Pourquoi les portfolios comptent

La plupart des ingénieurs comptent uniquement sur leur CV. Un portfolio vous donne un avantage considérable :

| Sans portfolio | Avec portfolio |
|----------------|----------------|
| « J'ai construit un moteur de recommandation » | Démo en direct + diagramme d'architecture + métriques |
| Le recruteur parcourt en 6 secondes | Le responsable du recrutement met en favori pour l'équipe |
| En compétition sur les mots-clés | En compétition sur les preuves |
| Unidimensionnel (texte uniquement) | Multidimensionnel (code + visuels + rédaction) |

<Callout type="info">
Un portfolio n'a pas besoin d'être un site web sophistiqué. Un profil GitHub bien organisé avec de bons READMEs est un portfolio. Cette leçon couvre les deux approches.
</Callout>

## Les trois piliers

### Pilier 1 : Profil GitHub

Votre GitHub est votre CV d'ingénieur. Optimisez-le :

**README de profil** (créez un dépôt avec votre nom d'utilisateur) :
```markdown
# Hi, I'm [Name] 👋

**Senior Engineer** specialising in [2-3 areas]

## 🔭 What I'm working on
- [Project 1] — one-line description
- [Project 2] — one-line description

## 📊 This Week's Focus
- [Current learning/building goal]

## 🛠️ Tech Stack
[Badges for your core technologies]

## 📈 Stats
[GitHub stats card — use github-readme-stats]
```

**Dépôts épinglés** (choisissez judicieusement — épinglez vos 6 meilleurs) :

| Type de dépôt | Pourquoi l'épingler |
|---------------|---------------------|
| **Projet de qualité production** | Montre que vous savez livrer |
| **Contribution open-source** | Montre que vous collaborez |
| **Implémentation de conception système** | Montre que vous pensez à grande échelle |
| **Enseignement/documentation** | Montre que vous communiquez |
| **Collection d'algorithmes** | Montre la profondeur technique |
| **Projet personnel avec des utilisateurs** | Montre la pensée produit |

### Pilier 2 : Approfondissements de projets

Chaque projet épinglé a besoin d'un **README exceptionnel** :

```markdown
# Project Name

One-sentence hook that explains the value.

## 🏗️ Architecture
[Architecture diagram — use Mermaid, Excalidraw, or draw.io]

## 🚀 Quick Start
[3-step setup that actually works]

## 📊 Performance
[Real metrics: response times, throughput, test coverage]

## 🧠 Key Design Decisions
[2-3 trade-offs you made and WHY]

## 📝 Blog Post / Write-up
[Link to detailed technical write-up]
```

<ThinkAboutIt>
Qu'est-ce qui fait qu'un README se démarque ? Pas la longueur — la clarté. Un responsable du recrutement devrait comprendre ce que vous avez construit, pourquoi c'est impressionnant et comment ça fonctionne en 30 secondes après avoir atterri sur la page.
</ThinkAboutIt>

### Pilier 3 : Rédaction technique

L'écriture est un **super-pouvoir d'ingénieur senior**. Commencez par :

1. **Blog Dev.to ou Hashnode** — écrivez sur ce que vous construisez
2. **Architecture Decision Records** — documentez les compromis dans vos projets
3. **Collection « Aujourd'hui j'ai appris »** — courts articles sur des découvertes intéressantes

**Formule d'article de blog qui fonctionne :**

```
Title: "How I [achieved result] with [technology]"

1. The Problem (2 paragraphs — make it relatable)
2. What I Tried First (and why it didn't work)
3. The Solution (with code + diagrams)
4. Results (metrics, before/after)
5. What I'd Do Differently (shows maturity)
```

## Idées de projets pour le portfolio

### Niveau 1 : Démontrer les fondamentaux

| Projet | Ce que ça démontre | Temps |
|--------|---------------------|-------|
| **Outil CLI** (par ex., git-stats) | Code propre, tests, packaging | 1 semaine |
| **API REST** avec authentification | Compétences backend, sensibilisation sécurité | 1 semaine |
| **Dashboard** avec données réelles | Frontend + visualisation de données | 1 semaine |

### Niveau 2 : Démontrer la pensée à grande échelle

| Projet | Ce que ça démontre | Temps |
|--------|---------------------|-------|
| **Raccourcisseur d'URL** avec analytics | Conception système, mise à l'échelle de base de données | 2 semaines |
| **Chat en temps réel** | WebSockets, files de messages | 2 semaines |
| **Bibliothèque de rate limiter** | Algorithmes, systèmes distribués | 1 semaine |

### Niveau 3 : Démontrer les compétences Senior+

| Projet | Ce que ça démontre | Temps |
|--------|---------------------|-------|
| **Bibliothèque open-source** | Conception d'API, documentation, communauté | Continu |
| **Série de blogs sur la conception système** | Communication technique, profondeur | 4 semaines |
| **Application full-stack avec CI/CD** | Livraison de bout en bout, DevOps | 3 semaines |

<Callout type="warning">
La qualité plutôt que la quantité. **3 excellents projets** valent mieux que 20 à moitié terminés. Chaque projet devrait avoir : du code fonctionnel, des tests, de la documentation et un README que vous seriez fier de montrer à un responsable du recrutement.
</Callout>

## Erreurs courantes

### ❌ Erreur 1 : Projets de tutoriel

> « J'ai suivi un tutoriel YouTube et construit une application de tâches »

Cela montre que vous savez suivre des instructions, pas résoudre des problèmes. **Solution :** Prenez n'importe quel projet de tutoriel et ajoutez 2-3 fonctionnalités qui n'étaient pas dans le tutoriel. Documentez POURQUOI vous les avez ajoutées.

### ❌ Erreur 2 : Pas de documentation

> Dépôt avec du code mais pas de README, pas de commentaires, pas de contexte

Si un responsable du recrutement ne peut pas comprendre votre projet en 30 secondes, il passera au suivant. **Solution :** Consacrez 20 % du temps de projet à la documentation.

### ❌ Erreur 3 : Dépôts abandonnés

> 15 dépôts, tous avec « Initial commit » et rien d'autre

Cela signale que vous commencez des choses mais ne les terminez pas. **Solution :** Archivez ou supprimez les dépôts inachevés. Ne montrez que le travail terminé.

### ❌ Erreur 4 : Pas de métriques

> « J'ai construit une API rapide »

Rapide comparé à quoi ? **Solution :** Ajoutez des benchmarks, des résultats de tests de charge, des scores Lighthouse — tout ce qui est quantifiable.

## Checklist de révision du portfolio

Avant de partager votre portfolio, vérifiez :

**Profil GitHub :**
- [ ] README de profil avec introduction claire
- [ ] 6 dépôts épinglés (votre meilleur travail)
- [ ] Historique de commits cohérent (carrés verts)
- [ ] Avatar professionnel et biographie

**Chaque projet :**
- [ ] README avec diagramme d'architecture
- [ ] Instructions de démarrage rapide fonctionnelles
- [ ] Tests avec couverture visible
- [ ] Au moins une « Décision clé » documentée
- [ ] Pas de secrets ni d'identifiants dans l'historique git

**Présence en ligne :**
- [ ] LinkedIn mis à jour avec les liens des projets
- [ ] Au moins 2 articles de blog techniques
- [ ] Profil Dev.to / Hashnode lié

**Prêt pour l'entretien :**
- [ ] Capable d'expliquer n'importe quel projet pendant 5 minutes sans notes
- [ ] Capable de discuter des compromis pour chaque décision clé
- [ ] Capable de décrire ce que vous amélioreriez avec plus de temps
- [ ] Avoir des métriques pour au moins 2 projets

## Plan d'action : Construisez votre portfolio en 30 jours

| Semaine | Objectif |
|---------|----------|
| **Semaine 1** | Configurer le README du profil GitHub + choisir 3 projets à mettre en avant |
| **Semaine 2** | Terminer le projet 1 avec README complet et tests |
| **Semaine 3** | Terminer le projet 2 + écrire le premier article de blog |
| **Semaine 4** | Terminer le projet 3 + créer un site portfolio (optionnel) |

<Callout type="info">
Vous n'avez pas besoin d'attendre que le portfolio soit « parfait » pour commencer à postuler. Publiez ce que vous avez, itérez, et laissez les retours réels guider vos améliorations.
</Callout>

---

## Pensée finale

Les ingénieurs qui obtiennent les meilleures offres ne sont pas toujours les plus qualifiés — ce sont ceux qui **rendent leurs compétences visibles**. Votre portfolio est le moyen d'y parvenir. Commencez aujourd'hui.
