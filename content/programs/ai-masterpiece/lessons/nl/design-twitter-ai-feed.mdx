# Ontwerp Twitter's AI Feed

Dit is een **afsluitende casestudy** — je combineert alles van AI Sketch tot AI Polish om een compleet AI-aangedreven sociale media feed te ontwerpen.

---

## De probleemstelling

> Ontwerp een systeem dat gepersonaliseerde tweets levert aan meer dan 500 miljoen gebruikers in real-time, waarbij content wordt gerangschikt op relevantie, recentheid en engagementpotentieel.

**Wat dit moeilijk maakt:**

- **Schaal:** 500 miljoen dagelijks actieve gebruikers, meer dan 500 miljoen tweets/dag
- **Latentie:** Feed moet binnen 200 ms laden
- **Versheid:** Nieuwe tweets verschijnen binnen seconden
- **Personalisatie:** Elke gebruiker ziet een unieke feed
- **Koude start:** Nieuwe gebruikers zonder historie

<Callout type="info">
Dit is exact het type vraag dat gesteld wordt bij senior+ interviews bij Meta, Google, Twitter/X en LinkedIn. Het test DSA, systeemontwerp EN ML-kennis tegelijkertijd.
</Callout>

## Fase 1: Datamodellering (DSA-fundamenten)

### Kernentiteiten

```
User { id, name, followers[], following[], interests[] }
Tweet { id, authorId, content, media[], timestamp, metrics }
Engagement { userId, tweetId, type: view|like|retweet|reply, timestamp }
```

### Belangrijke datastructuren

| Structuur | Doel | Waarom deze? |
|-----------|------|-------------|
| **Hash Map** | Gebruiker → profielzoekopdracht | O(1) toegang via userId |
| **Sorted Set** | Tijdlijnrangschikking | O(log n) invoegen + bereikquery's |
| **Bloom Filter** | "Al gezien" deduplicatie | Geheugenefficiënte lidmaatschapstest |
| **Inverted Index** | Contentzoeken | Koppelt tokens → tweet-ID's |
| **Graph (adjacency list)** | Sociale connecties | Volg/volger-relaties |

<ThinkAboutIt>
Waarom een Bloom Filter in plaats van een gewone HashSet voor "al gezien"-detectie? Met 500 miljoen gebruikers die elk meer dan 100 tweets/dag zien, zou een HashSet terabytes verbruiken. Een Bloom Filter gebruikt ongeveer 1% van dat geheugen met een acceptabel fout-positiefpercentage (ongeveer 1%).
</ThinkAboutIt>

### Fan-out-strategie: Hybride aanpak

**Fan-out on write** voor gebruikers met minder dan 10.000 volgers:
- Wanneer een gebruiker tweet, push naar alle volgerstijdlijnen
- Lage latentie bij lezen (tijdlijn is voorberekend)
- O(volgers) schrijfacties per tweet

**Fan-out on read** voor beroemdheden (meer dan 10.000 volgers):
- Niet pushen — samenvoegen op leesmoment
- Voorkomt schrijven naar miljoenen tijdlijnen
- Iets hogere leeslatentie

```
function handleNewTweet(tweet, author):
    if author.followerCount < CELEBRITY_THRESHOLD:
        // Fan-out on write
        for follower in author.followers:
            timelineCache.add(follower.id, tweet, score=tweet.timestamp)
    else:
        // Fan-out on read — store in celebrity tweets pool
        celebrityTweets.add(author.id, tweet)
```

<Callout type="warning">
De hybride aanpak is wat Twitter daadwerkelijk gebruikt. Pure fan-out-on-write schaalt niet voor accounts met miljoenen volgers (één tweet = miljoenen cache-schrijfacties).
</Callout>

## Fase 3: ML-rangschikkingsmodel

### Feature engineering

Het rangschikkingsmodel voorspelt: **P(gebruiker interacteert met tweet)**

| Feature-categorie | Voorbeelden |
|-------------------|-------------|
| **Gebruikersfeatures** | interesses, eerdere engagementpatronen, actieve uren |
| **Tweetfeatures** | leeftijd, mediatype, lengte, hashtags, populariteit auteur |
| **Kruisfeatures** | interactiegeschiedenis gebruiker-auteur, thematische overlap |
| **Contextfeatures** | tijdstip, apparaattype, verbindingssnelheid |

### Rangschikking in twee stappen

**Stap 1 — Kandidaatgeneratie (snel, breed):**
- Haal ongeveer 1.000 kandidaten op uit tijdlijncache + beroemdhedenpool
- Gebruik lichtgewicht model (logistische regressie) voor initiële scoring
- Latentiebudget: 50 ms

**Stap 2 — Fijnrangschikking (langzaam, precies):**
- Herrangschik de top 200 kandidaten
- Diep neuraal netwerk met attentiemechanisme
- Houd rekening met kruisfeatures en sequentiepatronen
- Latentiebudget: 100 ms

```
function rankFeed(userId, candidates):
    // Stage 1: Coarse ranking
    scored = candidates.map(tweet =>
        ({tweet, score: lightweightModel.predict(userId, tweet)})
    )
    top200 = scored.sortBy(s => -s.score).slice(0, 200)

    // Stage 2: Fine ranking
    reranked = deepModel.batchPredict(userId, top200)

    // Stage 3: Business rules (diversity, freshness boost)
    return applyBusinessRules(reranked)
```

### Diversiteit en versheid

Het rangschikkingsmodel alleen zou alleen virale content tonen. Pas post-rangschikkingsregels toe:

- **Thematische diversiteit:** Niet meer dan 3 opeenvolgende tweets over hetzelfde onderwerp
- **Auteursdiversiteit:** Verdeel tweets van dezelfde auteur
- **Versheidsboost:** Logaritmisch tijdsverval — nieuwere tweets krijgen een bonusscore
- **Exploratie:** 5% van de feedposities toont content buiten de bubbel van de gebruiker

## Fase 4: Omgaan met randgevallen

### Koude-startprobleem

Nieuwe gebruikers hebben geen engagementgeschiedenis. Oplossingen:

1. **Onboarding-interesses:** Vraag gebruikers om 3-5 onderwerpen te kiezen bij registratie
2. **Terugval op populaire content:** Toon trending tweets in geselecteerde onderwerpen
3. **Verkennen-exploiteren:** Agressief verkennen in de eerste 48 uur, dan exploiteren
4. **Demografische priors:** Gebruik locatie, taal, apparaat voor initiële signalen

### Real-time updates

```
// WebSocket connection for live updates
connection.onTweet(tweet => {
    if (isHighPriority(tweet)):  // Breaking news, close friend
        injectToFeed(tweet, position=TOP)
    else:
        showNewTweetsButton(count++)

```

### Misbruik en veiligheid

- **Contentmoderatie:** ML-classificator markeert schadelijke content vóór rangschikking
- **Snelheidsbeperking:** Token bucket per gebruiker (zie AI Craft — Rate Limiter)
- **Spamdetectie:** Op grafen gebaseerde analyse van volgpatronen + contentsimilariteit
