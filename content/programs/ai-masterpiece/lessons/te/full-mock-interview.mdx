## రౌండ్ 2: Coding (20 నిమిషాలు)

### సమస్య

> **TTL తో LRU Cache**
>
> `get(key)` మరియు `put(key, value, ttl)` operations కు support చేసే data structure డిజైన్ చేయండి.
> - `get(key)`: key ఉనికిలో ఉంటే మరియు expire కాకపోతే value return చేయండి. లేకపోతే -1 return చేయండి.
> - `put(key, value, ttl)`: సెకన్లలో time-to-live తో key-value pair ను set లేదా update చేయండి.
> - Cache capacity మించినప్పుడు, అత్యంత తక్కువ ఇటీవల ఉపయోగించిన non-expired entry ను తొలగించండి.

### దశ 1: స్పష్టం చేయండి (2 నిమిషాలు)

Code వ్రాయడానికి ముందు, అడగండి:

- "ఆశించిన capacity ఎంత?" → *10,000 entries అనుకోండి*
- "ఇది single-threaded నా?" → *అవును, ప్రస్తుతానికి*
- "సమయ precision ఏమిటి — సెకన్లు లేదా మిల్లీసెకన్లు?" → *సెకన్లు సరిపోతుంది*
- "Expired entries lazy గా clean అవ్వాలా లేదా eagerly?" → *Lazy ఆమోదయోగ్యం*

<ThinkAboutIt>
Interviewers మీరు ప్రశ్నలు అడగాలని కోరుకుంటారు. ఇది code లోకి దూకే ముందు requirements గురించి ఆలోచిస్తారని చూపిస్తుంది. 2-3 clarifying ప్రశ్నలు లక్ష్యంగా పెట్టుకోండి.
</ThinkAboutIt>

### దశ 2: విధానం (3 నిమిషాలు)

Code వ్రాయడానికి ముందు మీ విధానాన్ని వివరించండి:

```
"I'll combine a hash map for O(1) lookup with a doubly-linked list 
for O(1) LRU eviction — the classic LRU pattern from AI Sketch.

The twist is TTL: each node stores an expiry timestamp. On get(), 
I'll check if the entry has expired. On put(), I'll set 
expiry = now + ttl.

For eviction, I'll skip expired entries when walking from the 
tail (least recently used)."
```

### దశ 3: Code (10 నిమిషాలు)

```python
class Node:
    def __init__(self, key, value, expires_at):
        self.key = key
        self.value = value
        self.expires_at = expires_at
        self.prev = None
        self.next = None

class LRUCacheWithTTL:
    def __init__(self, capacity):
        self.capacity = capacity
        self.cache = {}  # key → Node
        self.head = Node(0, 0, 0)  # dummy head (most recent)
        self.tail = Node(0, 0, 0)  # dummy tail (least recent)
        self.head.next = self.tail
        self.tail.prev = self.head

    def _now(self):
        import time
        return int(time.time())

    def _remove(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_front(self, node):
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def _is_expired(self, node):
        return self._now() > node.expires_at

    def get(self, key):
        if key not in self.cache:
            return -1
        node = self.cache[key]
        if self._is_expired(node):
            self._remove(node)
            del self.cache[key]
            return -1
        # Move to front (most recently used)
        self._remove(node)
        self._add_to_front(node)
        return node.value

    def put(self, key, value, ttl):
        if key in self.cache:
            self._remove(self.cache[key])
            del self.cache[key]
        node = Node(key, value, self._now() + ttl)
        self.cache[key] = node
        self._add_to_front(node)
        # Evict if over capacity
        while len(self.cache) > self.capacity:
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]
```

### దశ 4: పరీక్ష (3 నిమిషాలు)

మౌఖికంగా ఒక test case ద్వారా నడవండి:

```
cache = LRUCacheWithTTL(2)
cache.put("a", 1, ttl=10)    # cache: [a]
cache.put("b", 2, ttl=5)     # cache: [b, a]
cache.get("a")                # returns 1, cache: [a, b]
cache.put("c", 3, ttl=10)    # evicts b (LRU), cache: [c, a]
# After 6 seconds...
cache.get("b")                # returns -1 (evicted)
```

### దశ 5: Follow-Up ప్రశ్నలు

Interviewer మరింత లోతుగా వెళ్తారు:

1. *"దీన్ని thread-safe ఎలా చేస్తారు?"*
   → Read-write lock: బహుళ readers, exclusive writer. లేదా `ConcurrentHashMap` + lock-free linked list ఉపయోగించండి.

2. *"Eager expiration cleanup ఎలా జోడిస్తారు?"*
   → `expires_at` ద్వారా sorted min-heap తో background thread. క్రమం తప్పకుండా expired entries తొలగించండి.

3. *"Time complexity ఏమిటి?"*
   → get: O(1), put: O(1) amortised. Space: O(capacity).

## రౌండ్ 3: System Design (25 నిమిషాలు)

### సమస్య

> **Real-time collaborative document editor డిజైన్ చేయండి** (Google Docs లాంటిది).
> Support చేయాలి: simultaneous editing, cursor tracking, version history, offline mode.

### Framework: AI Craft విధానాన్ని ఉపయోగించండి

**దశ 1 — Requirements (3 నిమిషాలు):**
- Functional: Real-time sync, conflict resolution, version history, sharing
- Non-functional: 100ms లోపు sync latency, 100K concurrent docs, 50 users/doc

**దశ 2 — High-Level Design (5 నిమిషాలు):**
```
Client (Browser) ←→ WebSocket Gateway ←→ Collaboration Service
                                               │
                                     ┌─────────┼─────────┐
                                     │         │         │
                                  CRDT      Version    Auth
                                  Engine    Store      Service
                                     │         │
                                  Document  Snapshot
                                  Store     Store
```

**దశ 3 — లోతైన అధ్యయనం: Conflict Resolution (10 నిమిషాలు):**

రెండు విధానాలు:
- **OT (Operational Transformation):** concurrent operations కు వ్యతిరేకంగా operations ను transform చేయండి. Google Docs ఉపయోగిస్తుంది. సరిగ్గా implement చేయడం సంక్లిష్టం.
- **CRDT (Conflict-free Replicated Data Types):** convergence యొక్క గణిత హామీ. కేంద్ర సర్వర్ అవసరం లేదు. Figma, Notion ఉపయోగిస్తాయి.

కొత్త systems కోసం CRDT సిఫార్సు — సరళమైన correctness హామీలు.

**దశ 4 — Trade-offs (5 నిమిషాలు):**
- OT vs CRDT: OT పరీక్షించబడింది కానీ సంక్లిష్టం; CRDT సరళం కానీ ఎక్కువ మెమరీ వాడుతుంది
- WebSocket vs SSE: ద్విదిశాత్మక కోసం WebSocket; read-heavy కోసం SSE
- Snapshot ఫ్రీక్వెన్సీ: ఎక్కువ snapshots = వేగవంతమైన లోడ్ కానీ ఎక్కువ storage

<Callout type="info">
Senior స్థాయిలో, interviewers "సరైన" సమాధానం కంటే మీ trade-off reasoning గురించి ఎక్కువ ఆసక్తి చూపిస్తారు. స్పష్టంగా చెప్పండి: "నేను Y కంటే X ఎంచుకుంటున్నాను ఎందుకంటే..."
</Callout>

## రౌండ్ 4: ప్రవర్తన (10 నిమిషాలు)

### ప్రశ్న 1: "మీ టీమ్‌తో మీరు విభేదించిన సందర్భం గురించి చెప్పండి."

AI Polish నుండి **STAR Framework** ఉపయోగించండి:

| భాగం | మీ సమాధానం |
|------|-----------|
| **Situation** | "మేము కొత్త product కోసం microservices మరియు monolith మధ్య నిర్ణయిస్తున్నాము..." |
| **Task** | "monolith-first తో ప్రారంభించాలని నేను నమ్మాను, కానీ టీమ్ microservices కోరుకుంది..." |
| **Action** | "నేను ఖర్చు అంచనాలతో తులనాత్మక పత్రం సిద్ధం చేసి proof of concept ప్రతిపాదించాను..." |
| **Result** | "మేము monolith-first ఎంచుకున్నాము, 6 వారాల ముందుగానే launch చేసాము, సంవత్సరం 2 లో microservices కు migrate చేసాము..." |

### ప్రశ్న 2: "ఒక వైఫల్యం గురించి చెప్పండి."

<ThinkAboutIt>
Interviewer ఆత్మ-అవగాహన మరియు వృద్ధి మానసికత పరీక్షిస్తున్నారు. నిజమైన వైఫల్యం ఎంచుకోండి (humble-brag కాదు). నిర్మాణం: ఏమి జరిగింది → ఎందుకు జరిగింది → ఏమి నేర్చుకున్నారు → ఏమి మార్చారు.
</ThinkAboutIt>

### ప్రశ్న 3: "మిమ్మల్ని ఎందుకు hire చేయాలి?"

నైపుణ్యాల జాబితా చెప్పకండి. మీ ప్రత్యేక కలయికను వారి నిర్దిష్ట అవసరాలకు అనుసంధానం చేయండి:
