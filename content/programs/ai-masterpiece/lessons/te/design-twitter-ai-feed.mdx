# Twitter యొక్క AI Feed డిజైన్ చేయండి

ఇది ఒక **capstone case study** — మీరు AI Sketch నుండి AI Polish వరకు అన్నింటినీ కలిపి ఒక పూర్తి AI-ఆధారిత సోషల్ మీడియా ఫీడ్ డిజైన్ చేస్తారు.

---

## సమస్య వివరణ

> 500 మిలియన్ కంటే ఎక్కువ వినియోగదారులకు real-time లో వ్యక్తిగతీకరించిన tweets అందించే వ్యవస్థను డిజైన్ చేయండి, content ను relevance, recency మరియు engagement potential ద్వారా rank చేస్తూ.

**దీన్ని కష్టతరం చేసేది:**

- **స్కేల్:** 500 మిలియన్ రోజువారీ క్రియాశీల వినియోగదారులు, రోజుకు 500 మిలియన్ కంటే ఎక్కువ tweets
- **Latency:** ఫీడ్ 200ms లోపు లోడ్ అవ్వాలి
- **తాజాదనం:** కొత్త tweets సెకన్లలో కనిపిస్తాయి
- **వ్యక్తిగతీకరణ:** ప్రతి వినియోగదారుడు ఒక ప్రత్యేక ఫీడ్ చూస్తాడు
- **Cold start:** చరిత్ర లేని కొత్త వినియోగదారులు

<Callout type="info">
ఇది Meta, Google, Twitter/X మరియు LinkedIn లో senior+ interviews లో అడిగే ఖచ్చితమైన ప్రశ్న రకం. ఇది ఒకేసారి DSA, system design మరియు ML జ్ఞానాన్ని పరీక్షిస్తుంది.
</Callout>

## దశ 1: డేటా మోడలింగ్ (DSA Foundations)

### ప్రధాన Entities

```
User { id, name, followers[], following[], interests[] }
Tweet { id, authorId, content, media[], timestamp, metrics }
Engagement { userId, tweetId, type: view|like|retweet|reply, timestamp }
```

### కీలక డేటా నిర్మాణాలు

| నిర్మాణం | ఉద్దేశ్యం | ఇది ఎందుకు? |
|----------|----------|------------|
| **Hash Map** | User → ప్రొఫైల్ లుకప్ | userId ద్వారా O(1) యాక్సెస్ |
| **Sorted Set** | టైమ్‌లైన్ ర్యాంకింగ్ | O(log n) ఇన్సర్ట్ + range queries |
| **Bloom Filter** | "ఇప్పటికే చూసిన" డీడుప్లికేషన్ | స్పేస్-సమర్థ సెట్ సభ్యత్వం |
| **Inverted Index** | కంటెంట్ సెర్చ్ | tokens → tweet IDs మ్యాప్ చేస్తుంది |
| **Graph (adjacency list)** | సామాజిక కనెక్షన్లు | Follow/follower సంబంధాలు |

<ThinkAboutIt>
"ఇప్పటికే చూసిన" గుర్తింపు కోసం సాధారణ HashSet బదులు Bloom Filter ఎందుకు? ప్రతి రోజు 100 కంటే ఎక్కువ tweets చూసే 500 మిలియన్ వినియోగదారులతో, HashSet టెరాబైట్ల మెమరీ వినియోగిస్తుంది. Bloom Filter ఆ మెమరీలో సుమారు 1% మాత్రమే వాడుతుంది, ఆమోదయోగ్యమైన false positive రేటు (సుమారు 1%) తో.
</ThinkAboutIt>

## దశ 2: సిస్టమ్ ఆర్కిటెక్చర్

### ఉన్నత-స్థాయి కంపోనెంట్లు

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│  Tweet       │────▶│  Fan-out     │────▶│  Timeline Cache  │
│  Ingestion   │     │  Service     │     │  (Redis Sorted   │
└─────────────┘     └──────────────┘     │   Sets)          │
                                          └────────┬────────┘
                                                   │
┌─────────────┐     ┌──────────────┐              ▼
│  User        │────▶│  Ranking     │     ┌─────────────────┐
│  Request     │     │  Service     │────▶│  Feed Response   │
└─────────────┘     └──────────────┘     └─────────────────┘
                           │
                    ┌──────┴──────┐
                    │  ML Model   │
                    │  Service    │
                    └─────────────┘
```

### Fan-Out వ్యూహం: హైబ్రిడ్ విధానం

**Fan-out on write** 10,000 కంటే తక్కువ followers ఉన్న వినియోగదారులకు:
- ఒక వినియోగదారుడు tweet చేసినప్పుడు, అన్ని follower timelines కు push చేయండి
- తక్కువ-latency reads (timeline ముందుగానే computed)
- ప్రతి tweet కు O(followers) writes

**Fan-out on read** celebrities (10,000 కంటే ఎక్కువ followers) కోసం:
- Push చేయవద్దు — చదివే సమయంలో merge చేయండి
- మిలియన్ల timelines లో వ్రాయడం నివారిస్తుంది
- కొంచెం ఎక్కువ read latency

```
function handleNewTweet(tweet, author):
    if author.followerCount < CELEBRITY_THRESHOLD:
        // Fan-out on write
        for follower in author.followers:
            timelineCache.add(follower.id, tweet, score=tweet.timestamp)
    else:
        // Fan-out on read — store in celebrity tweets pool
        celebrityTweets.add(author.id, tweet)
```

<Callout type="warning">
హైబ్రిడ్ విధానం Twitter నిజంగా ఉపయోగించేది. మిలియన్ల followers ఉన్న accounts కోసం pure fan-out-on-write స్కేల్ కాదు (ఒక tweet = మిలియన్ల cache writes).
</Callout>

## దశ 3: ML ర్యాంకింగ్ మోడల్

### Feature Engineering

ర్యాంకింగ్ మోడల్ అంచనా వేస్తుంది: **P(వినియోగదారుడు tweet తో engage అవుతాడు)**

| Feature వర్గం | ఉదాహరణలు |
|--------------|----------|
| **వినియోగదారు features** | ఆసక్తులు, గత engagement patterns, క్రియాశీల గంటలు |
| **Tweet features** | వయస్సు, మీడియా రకం, పొడవు, hashtags, రచయిత ప్రజాదరణ |
| **Cross features** | వినియోగదారు-రచయిత interaction చరిత్ర, టాపిక్ overlap |
| **Context features** | రోజు సమయం, పరికరం రకం, కనెక్షన్ వేగం |

### రెండు-దశల ర్యాంకింగ్

**దశ 1 — Candidate Generation (వేగవంతం, విస్తృతం):**
- Timeline cache + celebrity pool నుండి సుమారు 1,000 candidates తీసుకోండి
- ప్రారంభ స్కోరింగ్ కోసం తేలికపాటి మోడల్ (logistic regression) ఉపయోగించండి
- Latency బడ్జెట్: 50ms

**దశ 2 — Fine Ranking (నెమ్మదిగా, ఖచ్చితమైన):**
- టాప్ 200 candidates ను మళ్ళీ rank చేయండి
- Attention mechanism తో deep neural network
- Cross-features మరియు sequence patterns పరిగణించండి
- Latency బడ్జెట్: 100ms

```
function rankFeed(userId, candidates):
    // Stage 1: Coarse ranking
    scored = candidates.map(tweet =>
        ({tweet, score: lightweightModel.predict(userId, tweet)})
    )
    top200 = scored.sortBy(s => -s.score).slice(0, 200)

    // Stage 2: Fine ranking
    reranked = deepModel.batchPredict(userId, top200)

    // Stage 3: Business rules (diversity, freshness boost)
    return applyBusinessRules(reranked)
```

### వైవిధ్యం మరియు తాజాదనం

ర్యాంకింగ్ మోడల్ ఒక్కటే viral content మాత్రమే చూపిస్తుంది. పోస్ట్-ర్యాంకింగ్ నియమాలు వర్తింపజేయండి:

- **టాపిక్ వైవిధ్యం:** ఒకే టాపిక్ పై వరుసగా 3 కంటే ఎక్కువ tweets వద్దు
- **రచయిత వైవిధ్యం:** ఒకే రచయిత tweets ను వ్యాప్తి చేయండి
- **తాజాదనం బూస్ట్:** Logarithmic time decay — కొత్త tweets కు బోనస్ స్కోర్
- **అన్వేషణ:** ఫీడ్ slots లో 5% వినియోగదారు bubble బయటి content చూపిస్తాయి

## దశ 4: Edge Cases నిర్వహణ

### Cold Start సమస్య

కొత్త వినియోగదారులకు engagement చరిత్ర లేదు. పరిష్కారాలు:

1. **Onboarding ఆసక్తులు:** సైన్అప్ సమయంలో వినియోగదారులను 3-5 టాపిక్స్ ఎంచుకోమని అడగండి
2. **ప్రజాదరణ పొందిన content fallback:** ఎంచుకున్న టాపిక్స్ లో trending tweets చూపించండి
3. **Explore-exploit:** మొదటి 48 గంటలలో దూకుడుగా explore చేయండి, తర్వాత exploit చేయండి
4. **Demographic priors:** ప్రారంభ సంకేతాలకు స్థానం, భాష, పరికరం ఉపయోగించండి

### Real-Time అప్‌డేట్లు

```
// WebSocket connection for live updates
connection.onTweet(tweet => {
    if (isHighPriority(tweet)):  // Breaking news, close friend
        injectToFeed(tweet, position=TOP)
    else:
        showNewTweetsButton(count++)
})
```

### దుర్వినియోగం మరియు భద్రత

- **కంటెంట్ మోడరేషన్:** ML classifier ర్యాంకింగ్ కు ముందు హానికర content ను గుర్తిస్తుంది
- **Rate limiting:** ప్రతి వినియోగదారుడికి Token bucket (చూడండి AI Craft — Rate Limiter)
- **స్పామ్ డిటెక్షన్:** Follow patterns + content similarity యొక్క graph-based విశ్లేషణ

## దశ 5: ప్రవర్తనా చర్చ

నిజమైన interview లో, follow-up ప్రశ్నలు ఆశించండి:

<ThinkAboutIt>
1. "కొత్త ర్యాంకింగ్ మోడల్ మెరుగైనదని ఎలా కొలుస్తారు?" → Engagement metrics (CTR, గడిపిన సమయం, DAU retention) తో A/B testing
2. "ML మోడల్ పడిపోతే ఏం జరుగుతుంది?" → Chronological timeline కు graceful degradation
3. "రాజకీయ content bias ను ఎలా నిర్వహిస్తారు?" → పారదర్శకత నివేదికలు, వినియోగదారు నియంత్రణలు, వైవిధ్యమైన కమిటీ సమీక్ష
4. "ఈ సిస్టమ్ తో ఒక production incident గురించి చెప్పండి" → AI Polish నుండి STAR framework ఉపయోగించండి
</ThinkAboutIt>
