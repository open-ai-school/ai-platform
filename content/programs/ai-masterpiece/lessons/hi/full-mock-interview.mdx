# पूर्ण Mock Interview

यह पाठ एक **60 मिनट के senior engineer interview** का अनुकरण करता है — उस प्रकार का जो आपको एक शीर्ष tech कंपनी में सामना करना पड़ता। प्रत्येक अनुभाग पर ऐसे काम करें जैसे आप कमरे में हैं।

## राउंड 2: Coding (20 मिनट)

### समस्या

> **TTL के साथ LRU Cache**
>
> एक ऐसी data structure डिज़ाइन करें जो `get(key)` और `put(key, value, ttl)` operations को support करे।
> - `get(key)`: यदि key मौजूद है और expire नहीं हुई है तो value लौटाएँ। अन्यथा -1 लौटाएँ।
> - `put(key, value, ttl)`: key-value pair को सेकंडों में time-to-live के साथ सेट या अपडेट करें।
> - जब cache capacity से अधिक हो जाए, तो सबसे कम हाल ही में उपयोग की गई non-expired entry को हटाएँ।

### चरण 1: स्पष्ट करें (2 मिनट)

कोड लिखने से पहले, पूछें:

- "अपेक्षित capacity क्या है?" → *मान लें 10,000 entries*
- "क्या यह single-threaded है?" → *हाँ, अभी के लिए*
- "कौन सी समय सटीकता — सेकंड या मिलीसेकंड?" → *सेकंड ठीक है*
- "क्या expired entries को lazily clean किया जाना चाहिए या eagerly?" → *Lazy स्वीकार्य है*

<ThinkAboutIt>
Interviewers चाहते हैं कि आप प्रश्न पूछें। यह दिखाता है कि आप code में कूदने से पहले requirements के बारे में सोचते हैं। 2-3 clarifying प्रश्नों का लक्ष्य रखें।
</ThinkAboutIt>

### चरण 2: दृष्टिकोण (3 मिनट)

कोड लिखने से पहले अपना दृष्टिकोण समझाएँ:

```
"I'll combine a hash map for O(1) lookup with a doubly-linked list 
for O(1) LRU eviction — the classic LRU pattern from AI Sketch.

The twist is TTL: each node stores an expiry timestamp. On get(), 
I'll check if the entry has expired. On put(), I'll set 
expiry = now + ttl.

For eviction, I'll skip expired entries when walking from the 
tail (least recently used)."
```

### चरण 3: कोड (10 मिनट)

```python
class Node:
    def __init__(self, key, value, expires_at):
        self.key = key
        self.value = value
        self.expires_at = expires_at
        self.prev = None
        self.next = None

class LRUCacheWithTTL:
    def __init__(self, capacity):
        self.capacity = capacity
        self.cache = {}  # key → Node
        self.head = Node(0, 0, 0)  # dummy head (most recent)
        self.tail = Node(0, 0, 0)  # dummy tail (least recent)
        self.head.next = self.tail
        self.tail.prev = self.head

    def _now(self):
        import time
        return int(time.time())

    def _remove(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_front(self, node):
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def _is_expired(self, node):
        return self._now() > node.expires_at

    def get(self, key):
        if key not in self.cache:
            return -1
        node = self.cache[key]
        if self._is_expired(node):
            self._remove(node)
            del self.cache[key]
            return -1
        # Move to front (most recently used)
        self._remove(node)
        self._add_to_front(node)
        return node.value

    def put(self, key, value, ttl):
        if key in self.cache:
            self._remove(self.cache[key])
            del self.cache[key]
        node = Node(key, value, self._now() + ttl)
        self.cache[key] = node
        self._add_to_front(node)
        # Evict if over capacity
        while len(self.cache) > self.capacity:
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]
```

### चरण 4: परीक्षण (3 मिनट)

मौखिक रूप से एक test case पर चलें:

```
cache = LRUCacheWithTTL(2)
cache.put("a", 1, ttl=10)    # cache: [a]
cache.put("b", 2, ttl=5)     # cache: [b, a]
cache.get("a")                # returns 1, cache: [a, b]
cache.put("c", 3, ttl=10)    # evicts b (LRU), cache: [c, a]
# After 6 seconds...
cache.get("b")                # returns -1 (evicted)
```

### चरण 5: Follow-Up प्रश्न

Interviewer आगे बढ़ाएगा:

1. *"इसे thread-safe कैसे बनाएँगे?"*
   → Read-write lock: कई readers, exclusive writer। या `ConcurrentHashMap` + lock-free linked list का उपयोग करें।

2. *"Eager expiration cleanup कैसे जोड़ेंगे?"*
   → `expires_at` के अनुसार sorted min-heap के साथ background thread। समय-समय पर expired entries हटाएँ।

3. *"Time complexity क्या है?"*
   → get: O(1), put: O(1) amortised। Space: O(capacity)।

## राउंड 3: System Design (25 मिनट)

### समस्या

> **एक real-time collaborative document editor डिज़ाइन करें** (Google Docs जैसा)।
> Support करें: simultaneous editing, cursor tracking, version history, offline mode।

### Framework: AI Craft दृष्टिकोण का उपयोग करें

**चरण 1 — Requirements (3 मिनट):**
- Functional: Real-time sync, conflict resolution, version history, sharing
- Non-functional: 100ms से कम sync latency, 100K concurrent docs, 50 users/doc

**चरण 2 — High-Level Design (5 मिनट):**
```
Client (Browser) ←→ WebSocket Gateway ←→ Collaboration Service
                                               │
                                     ┌─────────┼─────────┐
                                     │         │         │
                                  CRDT      Version    Auth
                                  Engine    Store      Service
                                     │         │
                                  Document  Snapshot
                                  Store     Store
```

**चरण 3 — गहन अध्ययन: Conflict Resolution (10 मिनट):**

दो दृष्टिकोण:
- **OT (Operational Transformation):** concurrent operations के विरुद्ध operations को transform करें। Google Docs द्वारा उपयोग किया जाता है। सही ढंग से implement करना जटिल है।
- **CRDT (Conflict-free Replicated Data Types):** convergence की गणितीय गारंटी। कोई केंद्रीय सर्वर आवश्यक नहीं। Figma, Notion द्वारा उपयोग किया जाता है।

नए सिस्टमों के लिए CRDT की सिफारिश — सरल correctness गारंटी।

**चरण 4 — Trade-offs (5 मिनट):**
- OT vs CRDT: OT आजमाया हुआ है पर जटिल; CRDT सरल है पर अधिक मेमोरी उपयोग करता है
- WebSocket vs SSE: द्विदिशात्मक के लिए WebSocket; read-heavy के लिए SSE
- Snapshot आवृत्ति: अधिक snapshots = तेज़ लोड पर अधिक स्टोरेज

<Callout type="info">
Senior स्तर पर, interviewers को "सही" उत्तर से अधिक आपके trade-off reasoning की परवाह होती है। स्पष्ट रूप से बताएँ: "मैं Y के बजाय X चुन रहा हूँ क्योंकि..."
</Callout>

## राउंड 4: व्यवहार (10 मिनट)

### प्रश्न 1: "मुझे उस समय के बारे में बताएँ जब आप अपनी टीम से असहमत थे।"

AI Polish से **STAR Framework** का उपयोग करें:

| भाग | आपका उत्तर |
|-----|-----------|
| **Situation** | "हम एक नए product के लिए microservices और monolith के बीच निर्णय ले रहे थे..." |
| **Task** | "मैं मानता था कि हमें monolith-first शुरू करना चाहिए, लेकिन टीम microservices चाहती थी..." |
| **Action** | "मैंने लागत अनुमानों के साथ एक तुलना दस्तावेज़ तैयार किया और एक proof of concept प्रस्तावित किया..." |
| **Result** | "हमने monolith-first चुना, 6 सप्ताह पहले launch किया, और साल 2 में microservices में migrate किया..." |

### प्रश्न 2: "मुझे किसी विफलता के बारे में बताएँ।"

<ThinkAboutIt>
Interviewer आत्म-जागरूकता और विकास मानसिकता का परीक्षण कर रहा है। एक वास्तविक विफलता चुनें (humble-brag नहीं)। संरचना: क्या हुआ → क्यों हुआ → आपने क्या सीखा → आपने क्या बदला।
</ThinkAboutIt>

### प्रश्न 3: "हम आपको क्यों hire करें?"

कौशलों की सूची न दें। अपने अनूठे संयोजन को उनकी विशिष्ट आवश्यकताओं से जोड़ें:

```
"You're scaling [specific system]. I've done exactly that — 
I took [system] from [X] to [Y] scale, solving [specific challenge] 
along the way. I also bring [unique skill] that I notice your 
team is building toward."
```
