# Twitter का AI Feed डिज़ाइन करें

यह एक **capstone case study** है — आप AI Sketch से AI Polish तक सब कुछ मिलाकर एक संपूर्ण AI-संचालित सोशल मीडिया फ़ीड डिज़ाइन करेंगे।

---

## समस्या का विवरण

> एक ऐसा सिस्टम डिज़ाइन करें जो 500 मिलियन से अधिक उपयोगकर्ताओं को real-time में व्यक्तिगत tweets प्रदान करे, content को relevance, recency और engagement potential के आधार पर रैंक करे।

**इसे कठिन क्या बनाता है:**

- **स्केल:** 500 मिलियन दैनिक सक्रिय उपयोगकर्ता, प्रतिदिन 500 मिलियन से अधिक tweets
- **Latency:** फ़ीड 200ms से कम समय में लोड होना चाहिए
- **ताज़गी:** नए tweets सेकंडों में दिखाई देते हैं
- **वैयक्तिकरण:** प्रत्येक उपयोगकर्ता एक अनूठा फ़ीड देखता है
- **Cold start:** बिना इतिहास वाले नए उपयोगकर्ता

<Callout type="info">
यह ठीक उसी प्रकार का प्रश्न है जो Meta, Google, Twitter/X और LinkedIn में senior+ interviews में पूछा जाता है। यह एक साथ DSA, system design और ML ज्ञान का परीक्षण करता है।
</Callout>

## चरण 1: डेटा मॉडलिंग (DSA Foundations)

### मुख्य Entities

```
User { id, name, followers[], following[], interests[] }
Tweet { id, authorId, content, media[], timestamp, metrics }
Engagement { userId, tweetId, type: view|like|retweet|reply, timestamp }
```

### प्रमुख डेटा संरचनाएँ

| संरचना | उद्देश्य | यह क्यों? |
|--------|----------|-----------|
| **Hash Map** | User → प्रोफ़ाइल लुकअप | userId द्वारा O(1) एक्सेस |
| **Sorted Set** | टाइमलाइन रैंकिंग | O(log n) insert + range queries |
| **Bloom Filter** | "पहले देखा" डीडुप्लिकेशन | स्पेस-कुशल सेट सदस्यता |
| **Inverted Index** | कंटेंट सर्च | tokens → tweet IDs मैप करता है |
| **Graph (adjacency list)** | सामाजिक कनेक्शन | Follow/follower रिश्ते |

<ThinkAboutIt>
"पहले देखा" पहचान के लिए सामान्य HashSet के बजाय Bloom Filter क्यों? 500 मिलियन उपयोगकर्ताओं के साथ जो प्रतिदिन 100 से अधिक tweets देखते हैं, एक HashSet टेराबाइट्स मेमोरी खाएगा। एक Bloom Filter उस मेमोरी का लगभग 1% उपयोग करता है एक स्वीकार्य false positive दर (लगभग 1%) के साथ।
</ThinkAboutIt>

## चरण 2: सिस्टम आर्किटेक्चर

### उच्च-स्तरीय कंपोनेंट्स

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│  Tweet       │────▶│  Fan-out     │────▶│  Timeline Cache  │
│  Ingestion   │     │  Service     │     │  (Redis Sorted   │
└─────────────┘     └──────────────┘     │   Sets)          │
                                          └────────┬────────┘
                                                   │
┌─────────────┐     ┌──────────────┐              ▼
│  User        │────▶│  Ranking     │     ┌─────────────────┐
│  Request     │     │  Service     │────▶│  Feed Response   │
└─────────────┘     └──────────────┘     └─────────────────┘
                           │
                    ┌──────┴──────┐
                    │  ML Model   │
                    │  Service    │
                    └─────────────┘
```

### Fan-Out रणनीति: हाइब्रिड दृष्टिकोण

**Fan-out on write** 10,000 से कम followers वाले उपयोगकर्ताओं के लिए:
- जब कोई उपयोगकर्ता tweet करता है, सभी follower timelines में push करें
- कम-latency reads (timeline पहले से computed है)
- प्रति tweet O(followers) writes

**Fan-out on read** celebrities (10,000 से अधिक followers) के लिए:
- Push न करें — पढ़ने के समय merge करें
- लाखों timelines में लिखने से बचाता है
- पढ़ने की latency थोड़ी अधिक

```
function handleNewTweet(tweet, author):
    if author.followerCount < CELEBRITY_THRESHOLD:
        // Fan-out on write
        for follower in author.followers:
            timelineCache.add(follower.id, tweet, score=tweet.timestamp)
    else:
        // Fan-out on read — store in celebrity tweets pool
        celebrityTweets.add(author.id, tweet)
```

<Callout type="warning">
हाइब्रिड दृष्टिकोण वही है जो Twitter वास्तव में उपयोग करता है। शुद्ध fan-out-on-write लाखों followers वाले accounts के लिए स्केल नहीं करता (एक tweet = लाखों cache writes)।
</Callout>

## चरण 3: ML रैंकिंग मॉडल

### Feature Engineering

रैंकिंग मॉडल भविष्यवाणी करता है: **P(उपयोगकर्ता tweet के साथ engage करता है)**

| Feature श्रेणी | उदाहरण |
|---------------|---------|
| **उपयोगकर्ता features** | रुचियाँ, पिछले engagement patterns, सक्रिय घंटे |
| **Tweet features** | आयु, मीडिया प्रकार, लंबाई, hashtags, लेखक की लोकप्रियता |
| **Cross features** | उपयोगकर्ता-लेखक इंटरैक्शन इतिहास, विषय overlap |
| **Context features** | दिन का समय, डिवाइस प्रकार, कनेक्शन गति |

### दो-चरणीय रैंकिंग

**चरण 1 — Candidate Generation (तेज़, व्यापक):**
- Timeline cache + celebrity pool से लगभग 1,000 candidates प्राप्त करें
- प्रारंभिक स्कोरिंग के लिए हल्का मॉडल (logistic regression) उपयोग करें
- Latency बजट: 50ms

**चरण 2 — Fine Ranking (धीमा, सटीक):**
- शीर्ष 200 candidates को फिर से रैंक करें
- Attention mechanism के साथ deep neural network
- Cross-features और sequence patterns पर विचार करें
- Latency बजट: 100ms

```
function rankFeed(userId, candidates):
    // Stage 1: Coarse ranking
    scored = candidates.map(tweet =>
        ({tweet, score: lightweightModel.predict(userId, tweet)})
    )
    top200 = scored.sortBy(s => -s.score).slice(0, 200)

    // Stage 2: Fine ranking
    reranked = deepModel.batchPredict(userId, top200)

    // Stage 3: Business rules (diversity, freshness boost)
    return applyBusinessRules(reranked)
```

### विविधता और ताज़गी

अकेले रैंकिंग मॉडल केवल वायरल कंटेंट दिखाएगा। पोस्ट-रैंकिंग नियम लागू करें:

- **विषय विविधता:** एक ही विषय पर लगातार 3 से अधिक tweets नहीं
- **लेखक विविधता:** एक ही लेखक के tweets को फैलाएँ
- **ताज़गी बूस्ट:** Logarithmic time decay — नए tweets को बोनस स्कोर मिलता है
- **अन्वेषण:** फ़ीड के 5% slots उपयोगकर्ता के bubble से बाहर का कंटेंट दिखाते हैं

## चरण 4: Edge Cases को संभालना

### Cold Start समस्या

नए उपयोगकर्ताओं के पास कोई engagement इतिहास नहीं होता। समाधान:

1. **Onboarding रुचियाँ:** साइनअप के दौरान उपयोगकर्ताओं से 3-5 विषय चुनने को कहें
2. **लोकप्रिय कंटेंट fallback:** चयनित विषयों में trending tweets दिखाएँ
3. **Explore-exploit:** पहले 48 घंटों में आक्रामक रूप से explore करें, फिर exploit करें
4. **Demographic priors:** प्रारंभिक संकेतों के लिए स्थान, भाषा, डिवाइस का उपयोग करें

### Real-Time अपडेट

```
// WebSocket connection for live updates
connection.onTweet(tweet => {
    if (isHighPriority(tweet)):  // Breaking news, close friend
        injectToFeed(tweet, position=TOP)
    else:
        showNewTweetsButton(count++)
})
```

### दुरुपयोग और सुरक्षा

- **कंटेंट मॉडरेशन:** ML classifier रैंकिंग से पहले हानिकारक कंटेंट को चिह्नित करता है
- **Rate limiting:** प्रति उपयोगकर्ता Token bucket (देखें AI Craft — Rate Limiter)
- **स्पैम डिटेक्शन:** Follow patterns + content similarity का graph-based विश्लेषण

## चरण 5: व्यवहार संबंधी चर्चा

वास्तविक interview में, follow-up प्रश्नों की अपेक्षा करें:

<ThinkAboutIt>
1. "आप कैसे मापेंगे कि नया रैंकिंग मॉडल बेहतर है?" → Engagement metrics (CTR, बिताया गया समय, DAU retention) के साथ A/B testing
2. "जब ML मॉडल बंद हो जाए तो क्या होता है?" → Chronological timeline में graceful degradation
3. "राजनीतिक कंटेंट bias को कैसे संभालते हैं?" → पारदर्शिता रिपोर्ट, उपयोगकर्ता नियंत्रण, विविध समिति समीक्षा
4. "इस सिस्टम के साथ एक प्रोडक्शन incident के बारे में बताएँ" → AI Polish से STAR framework का उपयोग करें
</ThinkAboutIt>

## सारांश: आपने क्या प्रदर्शित किया

| कौशल क्षेत्र | यहाँ लागू |
|-------------|-----------|
| **DSA** (AI Sketch) | Hash maps, sorted sets, bloom filters, graphs |
| **Patterns** (AI Chisel) | Two pointers (समय सीमाओं के लिए sliding window), BFS (सोशल graph) |
| **System Design** (AI Craft) | वितरित आर्किटेक्चर, caching, fan-out रणनीतियाँ |
| **व्यवहार** (AI Polish) | Trade-off चर्चा, incident handling, नैतिक विचार |

एक **senior engineer** का उत्तर ऐसा दिखता है — यह algorithms को architecture और वास्तविक दुनिया की बाधाओं से जोड़ता है।
