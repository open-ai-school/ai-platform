# Engineering Portfolio

एक मजबूत portfolio *"यह व्यक्ति कौन है?"* और *"इस उम्मीदवार को fast-track करें"* के बीच का अंतर है। यह पाठ आपको सिखाता है कि एक ऐसा portfolio कैसे बनाएँ जो अलग दिखे।

---

## Portfolio क्यों महत्वपूर्ण हैं

अधिकांश engineers केवल अपने resume पर निर्भर करते हैं। एक portfolio आपको अनुचित लाभ देता है:

| Portfolio के बिना | Portfolio के साथ |
|-------------------|-----------------|
| "मैंने एक recommendation engine बनाया" | लाइव demo + architecture diagram + metrics |
| Recruiter 6 सेकंड में देखता है | Hiring manager टीम के लिए bookmark करता है |
| Keywords पर प्रतिस्पर्धा | सबूतों पर प्रतिस्पर्धा |
| एक-आयामी (केवल टेक्स्ट) | बहु-आयामी (कोड + विज़ुअल्स + लेखन) |

<Callout type="info">
Portfolio को एक शानदार वेबसाइट होने की ज़रूरत नहीं। मजबूत READMEs के साथ एक सुव्यवस्थित GitHub प्रोफ़ाइल एक portfolio है। यह पाठ दोनों दृष्टिकोणों को कवर करता है।
</Callout>

---

## तीन स्तंभ

### स्तंभ 1: GitHub प्रोफ़ाइल

आपका GitHub आपका engineering resume है। इसे optimize करें:

**प्रोफ़ाइल README** (अपने username से एक repo बनाएँ):
```markdown
# Hi, I'm [Name] 👋

**Senior Engineer** specialising in [2-3 areas]

## 🔭 What I'm working on
- [Project 1] — one-line description
- [Project 2] — one-line description

## 📊 This Week's Focus
- [Current learning/building goal]

## 🛠️ Tech Stack
[Badges for your core technologies]

## 📈 Stats
[GitHub stats card — use github-readme-stats]
```

**Pinned repositories** (समझदारी से चुनें — अपने 6 सर्वश्रेष्ठ pin करें):

| Repo प्रकार | Pin क्यों करें |
|-------------|----------------|
| **Production-quality project** | दिखाता है कि आप ship कर सकते हैं |
| **Open-source contribution** | दिखाता है कि आप collaborate करते हैं |
| **System design implementation** | दिखाता है कि आप scale पर सोचते हैं |
| **Teaching/documentation** | दिखाता है कि आप communicate करते हैं |
| **Algorithm collection** | तकनीकी गहराई दिखाता है |
| **उपयोगकर्ताओं वाला side project** | Product thinking दिखाता है |

### स्तंभ 2: Project Deep Dives

प्रत्येक pinned project को एक **शानदार README** की आवश्यकता है:

```markdown
# Project Name

One-sentence hook that explains the value.

## 🏗️ Architecture
[Architecture diagram — use Mermaid, Excalidraw, or draw.io]

## 🚀 Quick Start
[3-step setup that actually works]

## 📊 Performance
[Real metrics: response times, throughput, test coverage]

## 🧠 Key Design Decisions
[2-3 trade-offs you made and WHY]

## 📝 Blog Post / Write-up
[Link to detailed technical write-up]
```

<ThinkAboutIt>
README को क्या विशेष बनाता है? लंबाई नहीं — स्पष्टता। एक hiring manager को page पर आने के 30 सेकंड के भीतर समझ आ जाना चाहिए कि आपने क्या बनाया, यह प्रभावशाली क्यों है, और यह कैसे काम करता है।
</ThinkAboutIt>

### स्तंभ 3: तकनीकी लेखन

लेखन एक **senior engineer की महाशक्ति** है। शुरू करें:

1. **Dev.to या Hashnode blog** — जो बनाते हैं उसके बारे में लिखें
2. **Architecture Decision Records** — अपने projects में trade-offs दस्तावेज़ करें
3. **"आज मैंने सीखा" संग्रह** — दिलचस्प खोजों के बारे में छोटे पोस्ट

**ब्लॉग पोस्ट फ़ॉर्मूला जो काम करता है:**

```
Title: "How I [achieved result] with [technology]"

1. The Problem (2 paragraphs — make it relatable)
2. What I Tried First (and why it didn't work)
3. The Solution (with code + diagrams)
4. Results (metrics, before/after)
5. What I'd Do Differently (shows maturity)
```

## Portfolio Project विचार

### स्तर 1: मूल बातें प्रदर्शित करें

| Project | यह क्या प्रदर्शित करता है | समय |
|---------|--------------------------|-----|
| **CLI tool** (जैसे, git-stats) | साफ कोड, testing, packaging | 1 सप्ताह |
| **REST API** auth के साथ | Backend कौशल, सुरक्षा जागरूकता | 1 सप्ताह |
| **Dashboard** वास्तविक डेटा के साथ | Frontend + डेटा visualization | 1 सप्ताह |

### स्तर 2: Scale Thinking प्रदर्शित करें

| Project | यह क्या प्रदर्शित करता है | समय |
|---------|--------------------------|-----|
| **URL shortener** analytics के साथ | System design, database scaling | 2 सप्ताह |
| **Real-time chat** | WebSockets, message queues | 2 सप्ताह |
| **Rate limiter library** | Algorithms, distributed systems | 1 सप्ताह |

### स्तर 3: Senior+ कौशल प्रदर्शित करें

| Project | यह क्या प्रदर्शित करता है | समय |
|---------|--------------------------|-----|
| **Open-source library** | API design, documentation, community | चालू |
| **System design blog series** | तकनीकी संचार, गहराई | 4 सप्ताह |

<Callout type="warning">
गुणवत्ता मात्रा से ऊपर। **3 उत्कृष्ट projects** 20 आधे-अधूरे से बेहतर हैं। प्रत्येक project में होना चाहिए: काम करने वाला कोड, tests, documentation, और एक README जो आप hiring manager को दिखाने में गर्व महसूस करें।
</Callout>

## सामान्य गलतियाँ

### ❌ गलती 1: Tutorial Projects

> "मैंने एक YouTube tutorial follow किया और एक to-do app बनाई"

यह दिखाता है कि आप निर्देशों का पालन कर सकते हैं, समस्याएँ हल करना नहीं। **समाधान:** किसी भी tutorial project को लें और 2-3 ऐसी features जोड़ें जो tutorial में नहीं थीं। दस्तावेज़ करें कि आपने उन्हें क्यों जोड़ा।

### ❌ गलती 2: कोई Documentation नहीं

> कोड के साथ repository लेकिन कोई README नहीं, कोई comments नहीं, कोई context नहीं

यदि hiring manager आपके project को 30 सेकंड में नहीं समझ सकता, तो वे आगे बढ़ जाएँगे। **समाधान:** Project समय का 20% documentation पर खर्च करें।

### ❌ गलती 3: छोड़े गए Repos

> 15 repositories, सभी में "Initial commit" और कुछ नहीं

यह संकेत देता है कि आप चीज़ें शुरू करते हैं पर पूरी नहीं करते। **समाधान:** अधूरे repos को archive या delete करें। केवल पूर्ण कार्य दिखाएँ।

### ❌ गलती 4: कोई Metrics नहीं

> "मैंने एक तेज़ API बनाई"

किसकी तुलना में तेज़? **समाधान:** Benchmarks, load test results, Lighthouse scores जोड़ें — कुछ भी जो quantifiable हो।

## Portfolio Review Checklist

अपना portfolio साझा करने से पहले, सत्यापित करें:

**GitHub प्रोफ़ाइल:**
- [ ] स्पष्ट परिचय के साथ प्रोफ़ाइल README
- [ ] 6 pinned repos (आपका सर्वश्रेष्ठ कार्य)
- [ ] लगातार commit history (हरे चौकोर)
- [ ] पेशेवर avatar और bio

**प्रत्येक Project:**
- [ ] Architecture diagram के साथ README
- [ ] काम करने वाले quick-start निर्देश
- [ ] दृश्यमान coverage के साथ tests
- [ ] कम से कम एक "Key Decision" documented
- [ ] Git history में कोई secrets या credentials नहीं

**ऑनलाइन उपस्थिति:**
- [ ] Project links के साथ LinkedIn अपडेट
- [ ] कम से कम 2 तकनीकी blog posts
- [ ] Dev.to / Hashnode प्रोफ़ाइल linked

**Interview के लिए तैयार:**
- [ ] बिना notes के किसी भी project को 5 मिनट में समझा सकते हैं
- [ ] प्रत्येक key decision के trade-offs पर चर्चा कर सकते हैं
- [ ] अधिक समय मिलने पर क्या सुधारेंगे बता सकते हैं
- [ ] कम से कम 2 projects के लिए metrics हैं

## कार्य योजना: 30 दिनों में अपना Portfolio बनाएँ

| सप्ताह | लक्ष्य |
|--------|--------|
| **सप्ताह 1** | GitHub प्रोफ़ाइल README सेट करें + प्रदर्शित करने के लिए 3 projects चुनें |
| **सप्ताह 2** | पूर्ण README और tests के साथ Project 1 पूरा करें |
| **सप्ताह 3** | Project 2 पूरा करें + पहली blog post लिखें |
| **सप्ताह 4** | Project 3 पूरा करें + portfolio site बनाएँ (वैकल्पिक) |

<Callout type="info">
Portfolio के "perfect" होने तक apply करने के लिए इंतज़ार करने की ज़रूरत नहीं है। जो है वह publish करें, iterate करें, और वास्तविक feedback को अपने सुधारों का मार्गदर्शन करने दें।
</Callout>
