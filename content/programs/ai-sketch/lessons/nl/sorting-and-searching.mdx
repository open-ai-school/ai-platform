# Sorting & Searching

Elk zoekresultaat dat je ziet is **gesorteerd op relevantie**. Elke databasequery gebruikt **binair zoeken** op indexen. Deze algoritmen zijn overal — en ze zijn vaste kost bij sollicitatiegesprekken.

## Bubble Sort — De naïeve aanpak

> Wissel herhaaldelijk aangrenzende elementen als ze in de verkeerde volgorde staan.

**Doorloop** met `[5, 3, 8, 1]`:

| Ronde | Vergelijkingen | Resultaat |
|-------|---------------|-----------|
| 1 | 5→3 wissel, 5→8 overslaan, 8→1 wissel | [3, 5, 1, **8**] |
| 2 | 3→5 overslaan, 5→1 wissel | [3, 1, **5**, 8] |
| 3 | 3→1 wissel | [**1**, 3, 5, 8] |

**Tijd: O(n²) | Ruimte: O(1)**

<Callout type="info">
Bubble sort is alleen nuttig om te leren. Noem het in sollicitatiegesprekken om aan te tonen dat je weet waarom betere algoritmen bestaan — schakel dan onmiddellijk over naar merge sort of quick sort.
</Callout>

## Merge Sort — Verdeel en heers

> Splits de array recursief in tweeën en voeg gesorteerde helften weer samen.

**Het algoritme in 3 stappen:**

1. **Verdeel** — splits de array in twee helften
2. **Heers** — sorteer elke helft recursief
3. **Voeg samen** — combineer twee gesorteerde helften tot één gesorteerde array

**De samenvoegstap** is de sleutel — vergelijk de voorste elementen van elke helft:

| Links | Rechts | Keuze | Resultaat tot nu toe |
|-------|--------|-------|----------------------|
| **3**, 27, 38 | **9**, 43, 82 | 3 | [3] |
| 27, 38 | **9**, 43, 82 | 9 | [3, 9] |
| **27**, 38 | 43, 82 | 27 | [3, 9, 27] |
| 38 | 43, 82 | 38 | [3, 9, 27, 38] |
| — | 43, 82 | 43, 82 | [3, 9, 27, 38, 43, 82] |

```python
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])
    return merge(left, right)

def merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result
```

**Tijd: O(n log n) | Ruimte: O(n)**

<ThinkAboutIt>
Waarom O(n log n)? De array wordt log(n) keer gesplitst (halvering). Op elk niveau doen we O(n) werk aan het samenvoegen. Totaal: n werk × log(n) niveaus = O(n log n).
</ThinkAboutIt>

## Vergelijking van sorteeralgoritmen

| Algoritme | Beste | Gemiddeld | Slechtste | Ruimte | Stabiel? |
|-----------|-------|-----------|-----------|--------|----------|
| Bubble Sort | O(n) | O(n²) | O(n²) | O(1) | Ja |
| Merge Sort | O(n log n) | O(n log n) | O(n log n) | O(n) | Ja |
| Quick Sort | O(n log n) | O(n log n) | O(n²) | O(log n) | Nee |
| Heap Sort | O(n log n) | O(n log n) | O(n log n) | O(1) | Nee |

<Callout type="info">
**Stabiel** betekent dat gelijke elementen hun oorspronkelijke volgorde behouden. Dit is belangrijk bij het sorteren op meerdere sleutels — bijv. sorteren op naam, dan op leeftijd. Een stabiele sortering behoudt de naamvolgorde binnen groepen met dezelfde leeftijd.
</Callout>

## Binair zoeken — Het probleem halveren

> Vind een doel in een **gesorteerde** array door bij elke stap de helft van de elementen te elimineren.

**Vereiste:** De array MOET gesorteerd zijn.

**Doorloop** — zoek `23` in `[2, 5, 11, 15, 23, 37, 41, 56]`:

| Stap | Bereik | Midden | Vergelijking | Actie |
|------|--------|--------|-------------|-------|
| 1 | [2..56] | 15 | 23 > 15 | Zoek in rechterhelft |
| 2 | [23..56] | 37 | 23 &lt; 37 | Zoek in linkerhelft |
| 3 | [23] | 23 | 23 = 23 | **Gevonden op index 4!** |

**3 stappen** in plaats van 8 (lineaire scan). Voor 1 miljoen elementen: **20 stappen** in plaats van 1.000.000.

```python
def binary_search(nums, target):
    left, right = 0, len(nums) - 1
    while left <= right:
        mid = left + (right - left) // 2  # avoids overflow
        if nums[mid] == target:
            return mid
        elif nums[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1  # not found
```

**Tijd: O(log n) | Ruimte: O(1)**

<FunFact>
Binair zoeken is bedrieglijk lastig om correct te implementeren. Een studie van Jon Bentley toonde aan dat **90% van de professionele programmeurs** geen foutvrij binair zoekalgoritme kon schrijven. De meest voorkomende bug? Integer overflow in `(left + right) / 2` — daarom gebruiken we `left + (right - left) // 2`.
</FunFact>

## Varianten van binair zoeken

Binair zoeken is niet alleen voor "zoek een exacte waarde". Deze varianten komen voortdurend voor:

| Variant | Beschrijving | Belangrijke wijziging |
|---------|-------------|----------------------|
| **Ondergrens** | Eerste positie waar het element ingevoegd kan worden | `left` wanneer `nums[mid] >= target` → `right = mid` |
| **Bovengrens** | Positie na de laatste voorkomst | `left` wanneer `nums[mid] > target` → `right = mid` |
| **Zoeken in geroteerde array** | Gesorteerde array geroteerd op onbekend punt | Controleer eerst welke helft gesorteerd is |
| **Zoeken in antwoordruimte** | Vind min/max die aan een voorwaarde voldoet | Binair zoeken op het antwoord, niet op de array |

<ThinkAboutIt>
"Binair zoeken in de antwoordruimte" is een krachtige techniek. Voorbeeld: "Wat is de minimale capaciteit om pakketten in D dagen te verzenden?" Zoek binair tussen de minimale en maximale mogelijke capaciteit en controleer of elke kandidaat werkt. Dit transformeert een optimalisatieprobleem in een beslissingsprobleem.
</ThinkAboutIt>

## De AI-connectie: Rangschikking en ophaling

Sorteren en zoeken zijn fundamenteel voor hoe AI-systemen resultaten leveren:

**Zoekmachines (Google, Bing):**

- **Geïnverteerde indexen** — gesorteerde lijsten van document-ID's per zoekwoord
- **Binair zoeken** op deze indexen voor opzoekingen in minder dan een milliseconde
- **Samenvoegbewerkingen** om resultaten van meerdere zoekwoorden te combineren

**Aanbevelingssystemen:**

- Sorteer kandidaten op voorspelde score (relevantierangschikking)
- **Top-K selectie** — vind de K beste items zonder volledig te sorteren (gedeeltelijke sortering)
- **Approximate Nearest Neighbour** zoeken gebruikt boomstructuren vergelijkbaar met binair zoeken

**Database-queryoptimalisatie:**

- B-trees (een generalisatie van binaire zoekbomen) vormen de basis van elke SQL-index
- Queryplanners kiezen tussen sequentiële scan (O(n)) en indexscan (O(log n))

<FunFact>
Google verwerkt meer dan 8,5 miljard zoekopdrachten per dag. Elke query doorzoekt een index van honderden miljarden webpagina's. Zonder gesorteerde indexen en binair zoeken zou een enkele query minuten duren in plaats van milliseconden.
</FunFact>

## Complexiteitsoverzicht

| Bewerking | Ongesorteerde array | Gesorteerde array |
|-----------|--------------------|--------------------|
| Zoeken | O(n) | O(log n) |
| Invoegen | O(1) | O(n)* |
| Min/max vinden | O(n) | O(1) |
| k-de element vinden | O(n) | O(1) |

*Elementen moeten verschoven worden om de gesorteerde volgorde te behouden

<Callout type="info">
De afweging is duidelijk: **sorteren kost vooraf O(n log n)**, maar maakt O(log n) zoekopdrachten voor altijd mogelijk. Als je meer dan O(log n) keer zoekt, verdient het sorteren zichzelf terug.
</Callout>
