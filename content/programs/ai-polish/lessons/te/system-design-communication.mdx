# System Design కమ్యూనికేషన్ — బిగ్గరగా ఆలోచించండి

System design ఇంటర్వ్యూలు **మీరు ఎలా ఆలోచిస్తారో** పరీక్షిస్తాయి, కేవలం మీకు ఏమి తెలుసో కాదు. మీ సమాధానం యొక్క నిర్మాణం కంటెంట్ అంతే ముఖ్యం.

<Illustration src="/images/lessons/craft/design-interview-flow.svg" alt="System design interview timeline showing four phases: Clarify (5 min), High-Level (10 min), Deep Dive (20 min), Wrap Up (10 min)" caption="45-నిమిషాల system design ఇంటర్వ్యూ స్పష్టమైన సమయ కేటాయింపుతో నాలుగు విభిన్న దశలుగా విభజించబడింది" />

## 4-దశల ఫ్రేమ్‌వర్క్

### దశ 1: అవసరాలను స్పష్టం చేయండి (5 నిమిషాలు)

**వెంటనే డిజైన్ చేయడం మొదలుపెట్టకండి.** ముందుగా ప్రశ్నలు అడగండి.

- **ఫంక్షనల్ అవసరాలు** — వ్యవస్థ ఏమి చేస్తుంది?
- **నాన్-ఫంక్షనల్ అవసరాలు** — స్కేల్, latency, availability లక్ష్యాలు
- **పరిధి** — ఈ సంభాషణలో ఏది లోపల, ఏది బయట?
- **వినియోగదారులు** — ఎవరు వాడతారు? ఎంతమంది? Read/write నిష్పత్తి?

**ఎల్లప్పుడూ అడగాల్సిన ప్రశ్నలు:**
- "ఎంతమంది daily active users కోసం డిజైన్ చేయాలి?"
- "ఊహించిన read-to-write ratio ఏమిటి?"
- "Latency అవసరాలు ఉన్నాయా?"
- "Backend పై దృష్టి పెట్టాలా, లేదా client కూడా చేర్చాలా?"

<Callout type="info">
స్పష్టీకరణ ప్రశ్నలు అడగడం సీనియారిటీకి సంకేతం. జూనియర్ ఇంజనీర్లు నేరుగా పరిష్కారాల్లోకి దూకుతారు. సీనియర్ ఇంజనీర్లు ముందుగా సమస్యను నిర్వచిస్తారు.
</Callout>

### దశ 2: ఉన్నత-స్థాయి డిజైన్ (10 నిమిషాలు)

వివరాల ముందు పెద్ద చిత్రాన్ని గీయండి.

- **API డిజైన్** — 2-3 ప్రధాన endpoints నిర్వచించండి
- **ప్రధాన భాగాలు** — Load balancer, app servers, database, cache
- **డేటా మోడల్** — ముఖ్య entities మరియు సంబంధాలు (సరళంగా ఉంచండి)
- **డేటా ప్రవాహం** — ఒక request వ్యవస్థ గుండా ఎలా ప్రయాణిస్తుంది?

**మాట్లాడుతూ గీయండి:**
- సర్వీసులకు బాక్సులు
- డేటా ప్రవాహానికి బాణాలు
- టెక్నాలజీలకు లేబుల్స్
- ప్రతి లేయర్ వద్ద capacity నోట్ చేయండి

### దశ 3: లోతైన విశ్లేషణ (20 నిమిషాలు)

ఇక్కడ మీరు లోతు చూపిస్తారు. సమస్య ఆధారంగా **2-3 ప్రాంతాలు ఎంచుకోండి**.

- **Database డిజైన్** — Schema, indexing వ్యూహం, sharding విధానం
- **స్కేలింగ్** — Horizontal scaling, load balancing, auto-scaling
- **క్యాషింగ్** — ఏమి cache చేయాలి, TTL వ్యూహం, invalidation
- **మెసేజింగ్** — Async processing, event-driven నమూనాలు, queues
- **AI/ML pipeline** — Feature store, model serving, A/B testing (సంబంధిత అయితే)

<ThinkAboutIt>
"ఈ స్కేల్ వద్ద అవి సంభావ్య అడ్డంకులు కాబట్టి database layer మరియు caching పై దృష్టి పెట్టబోతున్నాను. మీరు వేరే ప్రాంతంలో లోతుగా వెళ్ళాలనుకుంటే చెప్పండి."
— ఈ ఒక్క వాక్యం నిర్మాణాత్మక ఆలోచన, ప్రాధాన్యీకరణ మరియు సహకారాన్ని చూపిస్తుంది.
</ThinkAboutIt>

### దశ 4: ముగింపు & ట్రేడ్-ఆఫ్‌లు (10 నిమిషాలు)

ఇంజనీరింగ్ పరిణతి ప్రదర్శిస్తూ బలంగా ముగించండి.

- **మీ డిజైన్‌ను 30 సెకన్లలో సంక్షిప్తం** చేయండి
- **మీరు చేసిన ట్రేడ్-ఆఫ్‌లు** మరియు ఎందుకో చర్చించండి
- **పరిమితులను గుర్తించండి** — 100x స్కేల్ వద్ద ఏమి విఫలమవుతుంది?
- **మెరుగుదలలు ప్రతిపాదించండి** — ఎక్కువ సమయం ఉంటే మీరు ఏమి జోడిస్తారు?
- **Edge cases పరిష్కరించండి** — వైఫల్య మోడ్‌లు, డేటా consistency

## వైట్‌బోర్డ్ & డయాగ్రమ్ ఉత్తమ పద్ధతులు

| చేయండి | చేయకండి |
|----|-------|
| స్థిరమైన ఆకారాలు వాడండి (సర్వీసులకు బాక్సులు, DBs కు సిలిండర్లు) | టెక్స్ట్ గోడ గీయండి |
| ప్రతి భాగం మరియు బాణానికి లేబుల్ పెట్టండి | లేబుల్ లేని గీతలను ఇంటర్వ్యూయర్ ట్రాక్ చేస్తారని భావించండి |
| లేయర్లు చూపించడానికి రంగు లేదా గ్రూపింగ్ వాడండి | అంతా ఒక మూలలో కుక్కండి |
| భాగాలపై capacity సంఖ్యలు రాయండి | స్కేల్‌ను అస్పష్టంగా వదిలేయండి |
| డిజైన్ మారినప్పుడు తుడిచి మళ్ళీ గీయండి | ఇప్పటికే ఉన్న డ్రాయింగ్‌లపై సవరణలు గీచండి |

## "Design X" ప్రశ్నలను ఎలా నిర్వహించాలి

సాధారణ ప్రశ్నలకు పునరావృతమయ్యే విధానం:

### "URL shortener డిజైన్ చేయండి"
- **స్పష్టం చేయండి:** Write-heavy లేదా read-heavy? Custom URLs? Analytics?
- **ముఖ్య నిర్ణయాలు:** Hashing vs counter, SQL vs NoSQL, caching layer
- **లోతుగా:** Collision handling, redirect latency, analytics pipeline

### "Chat application డిజైన్ చేయండి"
- **స్పష్టం చేయండి:** 1:1 లేదా group? Message persistence? Read receipts?
- **ముఖ్య నిర్ణయాలు:** WebSocket vs polling, message storage, presence service
- **లోతుగా:** Fan-out వ్యూహం, delivery guarantees, offline sync

### "ML recommendation system డిజైన్ చేయండి"
- **స్పష్టం చేయండి:** Recommendation రకం? Real-time లేదా batch? Cold start handling?
- **ముఖ్య నిర్ణయాలు:** Collaborative vs content-based, feature store, serving infra
- **లోతుగా:** Model training pipeline, A/B testing framework, feedback loops

<FunFact>
Google యొక్క system design rubric "Structured Approach" మరియు "Communication" కు మొత్తం స్కోర్‌లో 40% వెయిటేజ్ ఇస్తుంది. మీ డిజైన్ సాంకేతికంగా అసంపూర్ణంగా ఉన్నా, మీ ప్రక్రియ అద్భుతంగా ఉంటే మీరు ఉత్తీర్ణులు కాగలరు.
</FunFact>

## సమయ నిర్వహణ చిట్కాలు

- **ప్రతి దశకు మానసిక టైమర్ సెట్ చేయండి** — Clarify దశ Deep Dive సమయాన్ని తినకుండా చూడండి
- **ఆగిపోతే, చెప్పండి** — "X కోసం ఉత్తమ విధానం గురించి నాకు ఖచ్చితంగా తెలియదు, కానీ నేను దీన్ని ఎలా ఆలోచిస్తానో ఇదిగో..."
- **ప్రతి 10 నిమిషాలకు సరిచూసుకోండి** — "ఈ దిశ సరిగ్గా ఉందా? ఇక్కడ మరింత లోతుగా వెళ్ళమంటారా?"
- **ముగింపు కోసం 5 నిమిషాలు దాచుకోండి** — బాగా సంక్షేపించడం బలమైన చివరి ముద్ర వేస్తుంది

<Callout type="info">
నిజమైన టైమర్‌తో అభ్యాసం చేయండి. 45 నిమిషాల్లో "Design Twitter" కు సమాధానం చెప్తూ మిమ్మల్ని రికార్డ్ చేసుకోండి. తిరిగి చూడండి — మీరు ఎక్కడ నిర్మాణం కోల్పోతున్నారో లేదా setup పై ఎక్కువ సమయం ఖర్చు చేస్తున్నారో వెంటనే గమనిస్తారు.
</Callout>
