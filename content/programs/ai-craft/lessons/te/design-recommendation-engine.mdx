# Recommendation Engine డిజైన్ చేయండి

సిఫార్సులు Netflix లో ప్రజలు చూసే వాటిలో 80%, Amazon ఆదాయంలో 30%, మరియు మొత్తం TikTok అనుభవాన్ని నిర్ణయిస్తాయి. ఇక్కడ సిస్టమ్ డిజైన్ స్కేల్ వద్ద machine learning ను కలుస్తుంది.

<Callout type="info">
ఇది **ఆధునిక** సిస్టమ్ డిజైన్ ప్రశ్న. ఇది ML జ్ఞానం, డేటా పైప్‌లైన్ డిజైన్ మరియు పంపిణీ వ్యవస్థలను — అన్నింటినీ ఒక్కటిగా పరీక్షిస్తుంది. ఇంటర్వ్యూయర్లు దీన్ని ఇష్టపడతారు ఎందుకంటే ఒక్క „సరైన" సమాధానం లేదు.
</Callout>

## దశ 3: వినియోగదారు-అంశ ఇంటరాక్షన్ మ్యాట్రిక్స్

Collaborative filtering యొక్క పునాది **ఇంటరాక్షన్ మ్యాట్రిక్స్** — వినియోగదారు ప్రాధాన్యతల భారీ, విరళ పట్టిక.

### మ్యాట్రిక్స్

```
              Item_1  Item_2  Item_3  Item_4  Item_5  ...  Item_50K
User_1:       [  5      -      3       -       4     ...    -    ]
User_2:       [  -      4      -       5       -     ...    2    ]
User_3:       [  4      -      3       -       5     ...    -    ]
User_4:       [  -      5      -       4       -     ...    3    ]
...
User_200M:    [  -      -      4       -       -     ...    5    ]

Sparsity: ~99% empty (users only interact with tiny fraction of items)
```

### Matrix Factorization (Netflix ఎలా చేస్తుంది)

```python
# Decompose the sparse matrix into two dense matrices
# R (users x items) ≈ U (users x k) × V (k x items)
# k = latent factors (typically 50-200)

# User_1 embedding: [0.8, -0.2, 0.5, ...]  → "likes sci-fi, dislikes romance"
# Item_3 embedding: [0.7, -0.3, 0.4, ...]  → "is sci-fi, not romance"
# Predicted rating = dot_product(User_1, Item_3) = high score!

import numpy as np

def als_step(R, U, V, lambda_reg=0.1):
    """Alternating Least Squares — one optimization step"""
    # Fix V, solve for U
    for i in range(R.shape[0]):
        rated = R[i].nonzero()[0]
        if len(rated) == 0:
            continue
        V_rated = V[rated]
        R_rated = R[i, rated]
        U[i] = np.linalg.solve(
            V_rated.T @ V_rated + lambda_reg * np.eye(V.shape[1]),
            V_rated.T @ R_rated
        )
    return U
```

<Callout type="info">
ఆధునిక వ్యవస్థలు matrix factorization దాటి **deep learning మోడల్స్** కు వెళ్ళాయి — two-tower ఆర్కిటెక్చర్లు, transformers మరియు graph neural networks. కానీ matrix factorization అర్థం చేసుకోవడం అవసరం: ఇది ఇంటర్వ్యూయర్లు మీకు తెలుసని ఆశించే పునాది.
</Callout>

## దశ 4: పూర్తి సిస్టమ్ ఆర్కిటెక్చర్

ఇది పూర్తి ML పైప్‌లైన్ — ముడి డేటా నుండి 50K RPS వద్ద అందించే సిఫార్సుల వరకు.

<Illustration src="/images/lessons/craft/recommendation-arch.svg" alt="Full recommendation engine ML architecture showing data sources flowing through stream processing to feature store, with offline batch training path and online serving path including candidate generation, ML ranking, and re-ranking layers" caption="రెండు-మార్గ ఆర్కిటెక్చర్: Batch (ఆఫ్‌లైన్) మోడల్ శిక్షణ కోసం, రియల్-టైమ్ (ఆన్‌లైన్) serving కోసం. Feature Store రెండు ప్రపంచాలను అనుసంధానిస్తుంది, శిక్షణ మరియు inference కోసం స్థిరమైన features అందిస్తుంది." />

### ఆర్కిటెక్చర్ భాగాల లోతైన పరిశీలన

#### ఆఫ్‌లైన్ మార్గం (బ్యాచ్ ప్రాసెసింగ్)

| భాగం | ప్రయోజనం | సాంకేతికత | కేడెన్స్ |
|---|---|---|---|
| **Data Lake** | అన్ని ముడి ఈవెంట్‌లను నిల్వ చేయండి | S3, GCS | నిరంతరం |
| **Feature Engineering** | ముడి డేటా నుండి features గణించండి | Spark, dbt | గంటకు/రోజుకు |
| **మోడల్ శిక్షణ** | సిఫార్సు మోడల్స్ శిక్షణ ఇవ్వండి | PyTorch, TF on GPU | రోజువారీ |
| **మోడల్ రిజిస్ట్రీ** | మోడల్స్ వెర్షన్ మరియు డిప్లాయ్ చేయండి | MLflow, SageMaker | డిమాండ్ మీద |

#### ఆన్‌లైన్ మార్గం (రియల్-టైమ్ Serving)

| భాగం | ప్రయోజనం | సాంకేతికత | లేటెన్సీ |
|---|---|---|---|
| **Candidate Generation** | మిలియన్ల నుండి ~1000 కు తగ్గించండి | FAISS, Milvus (ANN) | 20ms కంటే తక్కువ |
| **Ranking Model** | అభ్యర్థులను స్కోర్ మరియు ర్యాంక్ చేయండి | TF Serving, Triton | 30ms కంటే తక్కువ |
| **Re-Ranking** | వ్యాపార నియమాలు, వైవిధ్యం వర్తింపజేయండి | కస్టమ్ లాజిక్ | 5ms కంటే తక్కువ |
| **Response Cache** | ఇటీవల సిఫార్సులను cache చేయండి | Redis | 2ms కంటే తక్కువ |

### రెండు-దశల పైప్‌లైన్

```
50,000 items in catalog
        |
   [Candidate Generation] — fast, approximate (ANN search)
        |
    1,000 candidates
        |
   [ML Ranking Model] — precise scoring with full features
        |
      Top 100 scored
        |
   [Re-Ranking Layer] — diversity, freshness, business rules
        |
    Final 20 shown to user
```

<ThinkAboutIt>
అన్ని 50K అంశాలను స్కోర్ చేయడానికి బదులు candidate generation + ranking లో ఎందుకు విభజించాలి? ఆలోచించండి: మీ ranking model ప్రతి అంశానికి 1ms తీసుకుంటే, 50K అంశాలను స్కోర్ చేయడం = 50 సెకన్లు. 1K అభ్యర్థులను స్కోర్ చేయడం = 1 సెకను. Candidate generation దశ approximate nearest neighbor (ANN) శోధన ఉపయోగించి నిర్వహించగల సమూహానికి చౌకగా ఫిల్టర్ చేస్తుంది.
</ThinkAboutIt>

## దశ 5: Feature Store

ML సిస్టమ్ డిజైన్ ఇంటర్వ్యూలలో మీరు చర్చించే **అత్యంత ముఖ్యమైన ఇన్ఫ్రాస్ట్రక్చర్ భాగం** Feature Store.

### Feature Stores ఎందుకు?

| లేకుండా సమస్య | Feature Store తో పరిష్కారం |
|---|---|
| Training/serving వ్యత్యాసం | రెండింటికీ ఒకే features |
| డూప్లికేట్ feature కోడ్ | సత్యానికి ఒకే మూలం |
| పాత features | ఆటోమేటిక్ తాజాదనం ట్రాకింగ్ |
| నెమ్మదైన feature గణన | ముందుగా గణించబడిన, cache చేసిన |

### Feature ఉదాహరణలు

```python
# User features (computed daily, stored in Feature Store)
user_features = {
    "user_id": "u_12345",
    "avg_session_duration_7d": 45.2,      # minutes
    "genre_affinity": [0.8, 0.2, 0.5],    # [action, comedy, drama]
    "active_hours": [18, 19, 20, 21],     # peak usage hours
    "device_type": "mobile",
    "account_age_days": 730,
    "total_items_consumed": 342,
}

# Item features (updated on catalog change)
item_features = {
    "item_id": "movie_789",
    "embedding": [0.12, -0.34, ...],       # 256-dim vector
    "genre": ["sci-fi", "action"],
    "release_year": 2024,
    "avg_rating": 4.2,
    "popularity_score_7d": 0.87,
    "content_length_min": 142,
}

# Context features (computed real-time)
context_features = {
    "time_of_day": "evening",
    "day_of_week": "saturday",
    "device": "smart_tv",
    "recent_interactions": ["movie_123", "movie_456"],
}
```

## దశ 6: A/B Testing Framework

కొలవలేని దాన్ని మెరుగుపరచలేము. సిఫార్సు వ్యవస్థలకు A/B testing అవసరం.

### ప్రయోగ సెటప్

```yaml
experiment:
  name: "two_tower_v2_vs_v1"
  traffic_split:
    control:   { model: "v1", weight: 90 }
    treatment: { model: "v2", weight: 10 }
  metrics:
    primary:   "click_through_rate"
    secondary: ["watch_time", "completion_rate", "diversity_score"]
  guardrails:
    - "unsubscribe_rate < 0.5%"
    - "latency_p99 < 150ms"
  duration: "14 days"
  min_sample_size: 100000
```

## దశ 7: రియల్-టైమ్ vs బ్యాచ్ సిఫార్సులు

### ఎప్పుడు ఏది ఉపయోగించాలి

| దృశ్యం | విధానం | ఎందుకు |
|---|---|---|
| **హోమ్‌పేజ్ „మీ కోసం"** | Batch + cache | గంటకు ఒకసారి గణన, వినియోగదారుడికి cache |
| **„మీరు ఇప్పుడే X చూసినందున"** | రియల్-టైమ్ | చివరి చర్యను ప్రతిబింబించాలి |
| **ఇమెయిల్ డైజెస్ట్‌లు** | Batch | రోజువారీగా బల్క్‌లో గణన |
| **„ఇప్పుడు ట్రెండింగ్"** | దాదాపు రియల్-టైమ్ | ప్రతి 5 నిమిషాలకు లోకప్రియత సమగ్రం |
| **శోధన ఫలితాలు** | రియల్-టైమ్ | సందర్భం-ఆధారిత, క్వెరీ-నిర్దిష్ట |

### రియల్-టైమ్ ఆర్కిటెక్చర్

```python
async def get_recommendations(user_id: str, context: dict) -> list:
    # 1. Check cache (< 2ms)
    cached = await redis.get(f"recs:{user_id}")
    if cached and not context.get("force_refresh"):
        return json.loads(cached)

    # 2. Fetch user features from Feature Store (< 5ms)
    user_features = await feature_store.get_online(user_id)

    # 3. Candidate generation via ANN search (< 20ms)
    user_embedding = user_features["embedding"]
    candidates = await faiss_index.search(user_embedding, k=1000)

    # 4. Fetch item features for candidates (< 10ms, batch)
    item_features = await feature_store.get_batch(candidates)

    # 5. Score with ML model (< 30ms)
    scores = await model_server.predict(user_features, item_features, context)

    # 6. Re-rank: apply diversity + business rules (< 5ms)
    results = re_rank(candidates, scores, diversity_weight=0.3)

    # 7. Cache results (TTL: 5 min)
    await redis.setex(f"recs:{user_id}", 300, json.dumps(results[:50]))

    return results[:20]  # Return top 20
    # Total latency: < 70ms
```

## AI లోతైన అధ్యయనం: ఆధునిక సిఫార్సు మోడల్స్

### Two-Tower Architecture (పరిశ్రమ ప్రమాణం)

```
User Tower                    Item Tower
    |                             |
[User ID]                   [Item ID]
[Demographics]              [Categories]
[History]                   [Metadata]
    |                             |
[Dense Layers]              [Dense Layers]
    |                             |
[User Embedding]      [Item Embedding]
    \                       /
     \                     /
      [Dot Product / Cosine]
              |
        [Relevance Score]
```

### రెండు టవర్లు ఎందుకు?

1. **అంశ embeddings ముందుగా గణించవచ్చు** — ఒకసారి గణించండి, మిలియన్ల సార్లు అందించండి
2. **అంశ embeddings పై ANN index** — మిల్లీసెకన్ కంటే తక్కువలో candidate పొందడం
3. **User tower రిక్వెస్ట్ సమయంలో నడుస్తుంది** — రియల్-టైమ్ సందర్భాన్ని సంగ్రహిస్తుంది
4. **స్వతంత్ర స్కేలింగ్** — అంశ index గంటకు అప్‌డేట్, వినియోగదారు features ప్రతి రిక్వెస్ట్‌కు అప్‌డేట్

### స్కేల్ వద్ద Embedding-ఆధారిత పొందడం

```python
# Pre-compute all item embeddings (offline, daily)
item_embeddings = model.item_tower(all_items)  # Shape: [50000, 256]
faiss_index = faiss.IndexFlatIP(256)            # Inner product index
faiss_index.add(item_embeddings)                # Build ANN index

# At serving time (online, per request)
user_embedding = model.user_tower(user_features)  # Shape: [1, 256]
distances, indices = faiss_index.search(user_embedding, k=1000)  # < 5ms
candidates = [items[i] for i in indices[0]]
```

<FunFact>
YouTube యొక్క సిఫార్సు వ్యవస్థ 800 మిలియన్ వీడియోలను ప్రాసెస్ చేస్తుంది మరియు 2 బిలియన్ లాగిన్ అయిన వినియోగదారులకు సేవలందిస్తుంది. వారి two-tower model వీడియో embeddings FAISS index లో శోధించడం ద్వారా 10ms కంటే తక్కువలో అభ్యర్థి వీడియోలను ఉత్పత్తి చేస్తుంది. Ranking model తర్వాత చూసిన చరిత్ర, శోధన చరిత్ర, జనసంఖ్యా మరియు రోజులో సమయం సహా 1000 కంటే ఎక్కువ features ఉపయోగించి ఈ అభ్యర్థులను స్కోర్ చేస్తుంది.
