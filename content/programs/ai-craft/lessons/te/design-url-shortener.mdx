# URL Shortener డిజైన్ చేయండి

ప్రతి సిస్టమ్ డిజైన్ ఇంటర్వ్యూ ఎక్కడో ప్రారంభమవుతుంది — మరియు URL shorteners మొదటి సమస్యగా పరిపూర్ణం. 5 నిమిషాల్లో అర్థం చేసుకునేంత సులభం, 45 నిమిషాలు చర్చించేంత లోతైనది.

<Callout type="info">
ఇది సిస్టమ్ డిజైన్ యొక్క „Hello World". ఈ నమూనాలో నేర్పరితనం సాధించండి మరియు తర్వాత వచ్చే ప్రతి డిజైన్ ప్రశ్నకు మీ వద్ద పదజాలం ఉంటుంది.
</Callout>

## దశ 1: అవసరాల సేకరణ

ఒక్క బాక్స్ కూడా గీయడానికి ముందు, **అవసరాలను స్పష్టం చేయండి**. ఇక్కడే ఇంటర్వ్యూయర్లు జూనియర్లను సీనియర్ల నుండి వేరు చేస్తారు.

### ఫంక్షనల్ అవసరాలు

| అవసరం | వివరాలు |
|---|---|
| **URL షార్టెన్ చేయండి** | పొడవైన URL ఇస్తే, చిన్న, ప్రత్యేక alias తిరిగి ఇవ్వండి |
| **రీడైరెక్ట్** | చిన్న URL యాక్సెస్ చేసినప్పుడు, అసలు URL కు రీడైరెక్ట్ చేయండి |
| **కస్టమ్ aliases** | ఐచ్ఛికంగా వినియోగదారులు తమ స్వంత షార్ట్ కోడ్ ఎంచుకోవడానికి అనుమతించండి |
| **గడువు** | URLs కు ఐచ్ఛిక TTL ఉండవచ్చు |
| **ఎనలిటిక్స్** | క్లిక్ కౌంట్, భౌగోళిక స్థానం, పరికరం ట్రాక్ చేయండి |

### నాన్-ఫంక్షనల్ అవసరాలు

| అవసరం | లక్ష్యం |
|---|---|
| **అందుబాటు** | 99.99% అప్‌టైమ్ (సంవత్సరానికి 53 నిమిషాల కంటే తక్కువ డౌన్‌టైమ్) |
| **లేటెన్సీ** | 50ms కంటే తక్కువలో రీడైరెక్ట్ (p99) |
| **స్కేల్** | నెలకు 100M కొత్త URLs, నెలకు 10 బిలియన్ రీడైరెక్ట్‌లు |
| **చదువు:రాయడం నిష్పత్తి** | 100:1 (చదువులు ప్రధానం) |
| **మన్నిక** | URLs ఎప్పటికీ కోల్పోకూడదు |

<ThinkAboutIt>
చదువు-రాయడం నిష్పత్తి ఎందుకు అంత ముఖ్యం? 100:1 నిష్పత్తి 1:1 నిష్పత్తితో పోలిస్తే మీ ఆర్కిటెక్చర్ నిర్ణయాలను ఎలా మారుస్తుంది? caching ఎక్కడ పెట్టాలి, ఎన్ని read replicas అవసరం, మరియు మీ write path కూడా వేగంగా ఉండాలా అనే దాని గురించి ఆలోచించండి.
</ThinkAboutIt>

ఇక్కడ పూర్తి సిస్టమ్ ఉంది — client నుండి database మరియు తిరిగి ప్రవాహాన్ని అధ్యయనం చేయండి.

|---|---|---|
| **Load Balancer** | API సర్వర్లలో ట్రాఫిక్ పంపిణీ | Nginx, ALB, Cloudflare |
| **API సర్వర్లు** | సృష్టి + రీడైరెక్ట్ లాజిక్ నిర్వహించండి | Node.js, Go, Java |
| **Cache** | రీడైరెక్ట్‌లు వేగవంతం చేయండి (హాట్ URLs) | Redis, Memcached |
| **Database** | శాశ్వత URL నిల్వ | PostgreSQL, DynamoDB |
| **Message Queue** | ఎనలిటిక్స్‌ను ప్రధాన మార్గం నుండి వేరు చేయండి | Kafka, SQS |
| **ఎనలిటిక్స్** | క్లిక్‌లు ట్రాక్ చేయండి, మెట్రిక్స్ సమగ్రం చేయండి | ClickHouse, BigQuery |

### API డిజైన్

```
POST /api/v1/shorten
Body: { "long_url": "https://example.com/...", "custom_alias": "my-link", "ttl": 86400 }
Response: { "short_url": "https://tiny.url/a1B2c3", "expires_at": "2025-01-01T00:00:00Z" }

GET /:short_code
Response: 301 Redirect → Location: https://example.com/...
```

<Callout type="info">
SEO ముఖ్యమైతే **301 (శాశ్వత రీడైరెక్ట్)** ఉపయోగించండి — బ్రౌజర్లు దీన్ని cache చేస్తాయి. ప్రతి క్లిక్ ట్రాక్ చేయాలంటే **302 (తాత్కాలిక రీడైరెక్ట్)** ఉపయోగించండి. చాలా URL shorteners ఎనలిటిక్స్ కోసం 302 ఉపయోగిస్తాయి.
</Callout>

## దశ 3: URL ఎన్‌కోడింగ్ వ్యూహం

సిస్టమ్ యొక్క హృదయం — చిన్న, ప్రత్యేక కోడ్‌లను ఎలా ఉత్పత్తి చేస్తాము?

<Illustration src="/images/lessons/craft/url-shortener-encoding.svg" alt="Base62 encoding flow showing long URL to hash function to Base62 conversion to short code with mathematical steps" caption="Base62 ఎన్‌కోడింగ్ సంఖ్యాత్మక ID ని సంక్షిప్త, URL-సురక్షిత స్ట్రింగ్‌గా మారుస్తుంది — 6 అక్షరాలు మాకు 56.8 బిలియన్ ప్రత్యేక URLs ఇస్తాయి" />

### విధానాల పోలిక

| విధానం | ప్రయోజనాలు | లోపాలు |
|---|---|---|
| **Auto-increment + Base62** | సరళం, ఘర్షణలు లేవు | ఊహించగలిగే, ఒకే వైఫల్య బిందువు |
| **MD5/SHA256 hash** | పంపిణీ చేసిన, సమన్వయకర్త అవసరం లేదు | ఘర్షణ ప్రమాదం, పొడవైన అవుట్‌పుట్ |
| **UUID** | సమన్వయం అవసరం లేదు | 36 అక్షరాలు — చిన్న URLs కు చాలా పొడవు |
| **ముందుగా ఉత్పత్తి చేసిన కీలు** | runtime గణన లేదు | కీ నిర్వహణ సేవ అవసరం |
| **Snowflake ID** | పంపిణీ చేసిన, క్రమబద్ధీకరించగల | మరింత సంక్లిష్టం, 64-bit IDs |

### విజయ వ్యూహం: కౌంటర్ + Base62

```python
import string

ALPHABET = string.digits + string.ascii_lowercase + string.ascii_uppercase  # 62 chars

def encode_base62(num: int) -> str:
    if num == 0:
        return ALPHABET[0]
    result = []
    while num > 0:
        result.append(ALPHABET[num % 62])
        num //= 62
    return ''.join(reversed(result))

# Example: encode_base62(2009215674) → "2bWPgs"
```

<FunFact>
bit.ly 6-7 అక్షర కోడ్‌లతో Base62 ఎన్‌కోడింగ్ ఉపయోగిస్తుంది. వారి స్కేల్ వద్ద (~600M సృష్టించిన లింక్‌లు), వారు 6-అక్షర కీ-స్పేస్‌లో 0.001% కంటే తక్కువ ఉపయోగించారు. మీ URL shortener పెరగడానికి చాలా స్థలం ఉంది!
</FunFact>

## దశ 4: డేటాబేస్ స్కీమా

```sql
CREATE TABLE urls (
    id          BIGSERIAL PRIMARY KEY,
    short_code  VARCHAR(7) UNIQUE NOT NULL,
    long_url    TEXT NOT NULL,
    created_at  TIMESTAMP DEFAULT NOW(),
    expires_at  TIMESTAMP,
    user_id     BIGINT,
    click_count BIGINT DEFAULT 0
);

CREATE INDEX idx_short_code ON urls(short_code);
CREATE INDEX idx_expires_at ON urls(expires_at) WHERE expires_at IS NOT NULL;
```

### Sharding వ్యూహం

100M+ URLs కోసం, ఒకే డేటాబేస్ సరిపోదు. `short_code` ద్వారా shard చేయండి:

| Shard | పరిధి | ఉదాహరణ |
|---|---|---|
| Shard 0 | short_code 0-9 తో మొదలవుతుంది | `3xK9mP` |
| Shard 1 | short_code a-m తో మొదలవుతుంది | `aB7nQr` |
| Shard 2 | short_code n-z తో మొదలవుతుంది | `pL2wXy` |
| Shard 3 | short_code A-Z తో మొదలవుతుంది | `Rt5vKm` |

<ThinkAboutIt>
`user_id` కు బదులు `short_code` ద్వారా ఎందుకు shard చేయాలి? చదువు మార్గం గురించి ఆలోచించండి — ఎవరైనా చిన్న URL పై క్లిక్ చేసినప్పుడు, మీ వద్ద కేవలం short_code మాత్రమే ఉంటుంది. వేరే ఏదీ చూడకుండా నేరుగా సరైన shard కు రూట్ చేయాలి.
</ThinkAboutIt>

## దశ 5: Cache పొర

100:1 చదువు-రాయడం నిష్పత్తితో, caching కీలకం. ఇక్కడ cache-aside నమూనా ఉంది:

### Cache Hit ప్రవాహం (సాధారణ మార్గం)
1. రిక్వెస్ట్ వస్తుంది: `GET /a1B2c3`
2. Redis తనిఖీ: `GET url:a1B2c3`
3. Cache HIT → `long_url` తిరిగి ఇవ్వండి → 301 రీడైరెక్ట్
4. మొత్తం లేటెన్సీ: **~2ms**

### Cache Miss ప్రవాహం
1. రిక్వెస్ట్ వస్తుంది: `GET /a1B2c3`
2. Redis తనిఖీ: `GET url:a1B2c3` → MISS
3. డేటాబేస్ క్వెరీ: `SELECT long_url FROM urls WHERE short_code = 'a1B2c3'`
4. Cache కు రాయండి: `SET url:a1B2c3 "https://..." EX 86400`
5. `long_url` తిరిగి ఇవ్వండి → 301 రీడైరెక్ట్
6. మొత్తం లేటెన్సీ: **~15ms**

### Cache కాన్ఫిగరేషన్

```
Cache size:    ~20% of total URLs (hot set)
Eviction:      LRU (Least Recently Used)
TTL:           24 hours (balance freshness vs hit rate)
Expected hit rate: 80-90% (Pareto: 20% of URLs get 80% of traffic)
```

<Callout type="info">
**Cache వార్మింగ్ వ్యూహం**: deployment సమయంలో అత్యధికంగా క్లిక్ చేయబడిన టాప్ 1000 URLs ను cache లో ముందుగా లోడ్ చేయండి. ఇది restart తర్వాత cache misses వరదను నివారిస్తుంది.
</Callout>

### క్షితిజ స్కేలింగ్ చెక్‌లిస్ట్

| పొర | వ్యూహం | నోట్స్ |

| **ID Generation** | పంపిణీ చేసిన: range-based లేదా Snowflake | సింగిల్ కౌంటర్ అడ్డంకిని నివారించండి |

### హాట్ URLs నిర్వహణ

కొన్ని URLs వైరల్ అవుతాయి — నిమిషాల్లో మిలియన్ల క్లిక్‌లు. వ్యూహాలు:

1. **బహుళ-స్థాయి cache**: L1 (స్థానిక ఇన్-మెమరీ) → L2 (Redis) → DB
2. **Rate limiting**: Backend ను వరద నుండి రక్షించండి
3. **CDN**: Edge వద్ద రీడైరెక్ట్‌లు cache చేయండి (ఎనలిటిక్స్‌తో జాగ్రత్తగా)

## AI కోణం: ML-ఆధారిత URL Shortening

ఆధునిక URL shorteners రెండు కీలక రంగాల్లో AI ఉపయోగిస్తాయి:

### 1. Spam/Phishing గుర్తింపు

```python
# Features for ML spam classifier
features = {
    "url_length": len(long_url),
    "has_ip_address": bool(re.match(r'\d+\.\d+\.\d+\.\d+', domain)),
    "subdomain_count": domain.count('.'),
    "uses_https": long_url.startswith('https'),
    "domain_age_days": get_domain_age(domain),
    "similar_to_known_brand": brand_similarity_score(domain),
    "contains_suspicious_keywords": check_keywords(long_url),
}
# Model: Random Forest or BERT-based classifier
# Precision > 99.5% needed to avoid false positives
```

### 2. క్లిక్ ప్రెడిక్షన్ (Cache Warming)

ML మోడల్స్ ఏ URLs ట్రాఫిక్ స్పైక్‌లు పొందుతాయో అంచనా వేస్తాయి, ముందస్తు cache warming ను అనుమతిస్తాయి:

- **ఇన్‌పుట్ features**: రోజులో సమయం, మూల ప్లాట్‌ఫార్మ్ (Twitter/Reddit), కంటెంట్ వర్గం, చారిత్రక నమూనాలు
- **అవుట్‌పుట్**: తదుపరి గంట కోసం అంచనా వేసిన క్లిక్ వాల్యూమ్
- **చర్య**: అంచనా వేసిన వైరల్ URLs కోసం cache ముందుగా వేడి చేయండి

<FunFact>
Bitly నెలకు 10 బిలియన్+ క్లిక్‌లను ప్రాసెస్ చేస్తుంది మరియు దుర్భావన URLs ను రియల్-టైమ్‌లో గుర్తించడానికి ML ఉపయోగిస్తుంది. వారి మోడల్ చిన్న URL సృష్టించబడిన సెకన్లలో phishing ప్రయత్నాలను పట్టుకుంటుంది — మొదటి బాధితుడు క్లిక్ చేయడానికి ముందే.
</FunFact>

## ఇంటర్వ్యూ చెక్‌లిస్ట్

మీ డిజైన్‌ను స్వయం-మూల్యాంకనం చేయడానికి దీన్ని ఉపయోగించండి:

- [ ] అవసరాలు స్పష్టం చేశారు (ఫంక్షనల్ + నాన్-ఫంక్షనల్)
- [ ] స్కేల్ అంచనా వేశారు (URLs/నెల, చదువులు/సెకను, స్టోరేజ్)
- [ ] శుభ్రమైన API డిజైన్ చేశారు (REST, సరైన status codes)
- [ ] సహేతుకతతో ఎన్‌కోడింగ్ వ్యూహం ఎంచుకున్నారు
- [ ] ఇండెక్సింగ్‌తో డేటాబేస్ స్కీమా డిజైన్ చేశారు
- [ ] స్పష్టమైన hit/miss ప్రవాహాలతో caching జోడించారు
- [ ] Sharding వ్యూహం చర్చించారు
- [ ] ఒకే వైఫల్య బిందువులను పరిష్కరించారు
- [ ] పర్యవేక్షణ మరియు అలర్టింగ్ ప్రస్తావించారు
- [ ] AI కోణం జోడించారు (spam గుర్తింపు, ప్రెడిక్షన్)

<ThinkAboutIt>
URLs **తొలగించగలిగేవిగా** (GDPR సమ్మతి) ఉండాలంటే మీరు ఈ డిజైన్‌ను ఎలా మార్చుతారు? Cache invalidation, డేటాబేస్‌లో soft deletes vs hard deletes, మరియు తొలగించిన URL కోసం రీడైరెక్ట్ రిక్వెస్ట్‌ను ఎలా నిర్వహిస్తారో ఆలోచించండి.
</ThinkAboutIt>
