# URL Shortener डिज़ाइन करें

हर सिस्टम डिज़ाइन इंटरव्यू कहीं से शुरू होता है — और URL shorteners पहली समस्या के रूप में उत्तम हैं। 5 मिनट में समझने के लिए पर्याप्त सरल, 45 मिनट तक चर्चा करने के लिए पर्याप्त गहरा।

<Callout type="info">
यह सिस्टम डिज़ाइन का „Hello World" है। इस पैटर्न में महारत हासिल करें और आपके पास आगे आने वाले हर डिज़ाइन प्रश्न के लिए शब्दावली होगी।
</Callout>

## चरण 1: आवश्यकताएँ एकत्र करना

एक भी बॉक्स बनाने से पहले, **आवश्यकताएँ स्पष्ट करें**। यहीं इंटरव्यूअर जूनियर और सीनियर के बीच अंतर करते हैं।

### कार्यात्मक आवश्यकताएँ

| आवश्यकता | विवरण |
|---|---|
| **URL छोटा करें** | एक लंबी URL दी गई हो, एक छोटा, अद्वितीय उपनाम लौटाएँ |
| **रीडायरेक्ट** | जब छोटी URL एक्सेस की जाए, मूल URL पर रीडायरेक्ट करें |
| **कस्टम उपनाम** | वैकल्पिक रूप से उपयोगकर्ताओं को अपना शॉर्ट कोड चुनने दें |
| **समाप्ति** | URLs में एक वैकल्पिक TTL हो सकता है |
| **एनालिटिक्स** | क्लिक गणना, भौगोलिक स्थान, डिवाइस ट्रैक करें |

### गैर-कार्यात्मक आवश्यकताएँ

| आवश्यकता | लक्ष्य |
|---|---|
| **उपलब्धता** | 99.99% अपटाइम (प्रति वर्ष 53 मिनट से कम डाउनटाइम) |
| **विलंबता** | 50ms से कम में रीडायरेक्ट (p99) |
| **पैमाना** | प्रति माह 100M नई URLs, प्रति माह 10 बिलियन रीडायरेक्ट |
| **पठन:लेखन अनुपात** | 100:1 (पठन प्रमुख) |
| **स्थायित्व** | URLs कभी खोनी नहीं चाहिए |

<ThinkAboutIt>
पठन-से-लेखन अनुपात इतना महत्वपूर्ण क्यों है? 100:1 का अनुपात 1:1 अनुपात की तुलना में आपके आर्किटेक्चर निर्णयों को कैसे बदलता है? सोचें कि आप caching कहाँ रखेंगे, कितनी read replicas चाहिए, और क्या आपके write path को तेज़ होने की भी ज़रूरत है।
</ThinkAboutIt>

## चरण 2: उच्च-स्तरीय आर्किटेक्चर

यहाँ पूरा सिस्टम है — client से database और वापस के प्रवाह का अध्ययन करें।

<Illustration src="/images/lessons/craft/url-shortener-arch.svg" alt="URL Shortener system architecture showing Client, Load Balancer, API Servers, Redis Cache, Database, and Analytics pipeline with ML spam detection" caption="पूर्ण URL Shortener आर्किटेक्चर — पठन/लेखन पथ (ऊपर) और async एनालिटिक्स पाइपलाइन (नीचे) के बीच विभाजन पर ध्यान दें" />

### मुख्य घटक

| घटक | उद्देश्य | तकनीक |
|---|---|---|
| **Load Balancer** | API सर्वरों में ट्रैफिक वितरित करें | Nginx, ALB, Cloudflare |
| **API सर्वर** | बनाने + रीडायरेक्ट लॉजिक संभालें | Node.js, Go, Java |
| **Cache** | रीडायरेक्ट तेज़ करें (लोकप्रिय URLs) | Redis, Memcached |
| **Database** | स्थायी URL संग्रहण | PostgreSQL, DynamoDB |
| **Message Queue** | एनालिटिक्स को मुख्य पथ से अलग करें | Kafka, SQS |
| **एनालिटिक्स** | क्लिक ट्रैक करें, मेट्रिक्स एकत्र करें | ClickHouse, BigQuery |

### API डिज़ाइन

```
POST /api/v1/shorten
Body: { "long_url": "https://example.com/...", "custom_alias": "my-link", "ttl": 86400 }
Response: { "short_url": "https://tiny.url/a1B2c3", "expires_at": "2025-01-01T00:00:00Z" }

GET /:short_code
Response: 301 Redirect → Location: https://example.com/...
```

<Callout type="info">
**301 (स्थायी रीडायरेक्ट)** का उपयोग करें यदि SEO महत्वपूर्ण है — ब्राउज़र इसे कैश करते हैं। **302 (अस्थायी रीडायरेक्ट)** का उपयोग करें यदि आपको हर क्लिक ट्रैक करना है। अधिकांश URL shorteners एनालिटिक्स के लिए 302 का उपयोग करते हैं।
</Callout>

## चरण 3: URL एन्कोडिंग रणनीति

सिस्टम का हृदय — हम छोटे, अद्वितीय कोड कैसे उत्पन्न करते हैं?

<Illustration src="/images/lessons/craft/url-shortener-encoding.svg" alt="Base62 encoding flow showing long URL to hash function to Base62 conversion to short code with mathematical steps" caption="Base62 एन्कोडिंग एक संख्यात्मक ID को एक संक्षिप्त, URL-सुरक्षित स्ट्रिंग में बदलता है — 6 अक्षर हमें 56.8 बिलियन अद्वितीय URLs देते हैं" />

### दृष्टिकोण तुलना

| दृष्टिकोण | फायदे | नुकसान |
|---|---|---|
| **Auto-increment + Base62** | सरल, कोई टकराव नहीं | पूर्वानुमानित, एकल विफलता बिंदु |
| **MD5/SHA256 hash** | वितरित, कोई समन्वयक नहीं | टकराव जोखिम, लंबा आउटपुट |
| **UUID** | कोई समन्वय आवश्यक नहीं | 36 अक्षर — छोटी URLs के लिए बहुत लंबा |
| **पूर्व-उत्पन्न कुंजियाँ** | कोई runtime गणना नहीं | कुंजी प्रबंधन सेवा चाहिए |
| **Snowflake ID** | वितरित, क्रमबद्ध | अधिक जटिल, 64-bit IDs |

### विजेता रणनीति: काउंटर + Base62

```python
import string

ALPHABET = string.digits + string.ascii_lowercase + string.ascii_uppercase  # 62 chars

def encode_base62(num: int) -> str:
    if num == 0:
        return ALPHABET[0]
    result = []
    while num > 0:
        result.append(ALPHABET[num % 62])
        num //= 62
    return ''.join(reversed(result))

# Example: encode_base62(2009215674) → "2bWPgs"
```

<FunFact>
bit.ly 6-7 अक्षर के कोड के साथ Base62 एन्कोडिंग का उपयोग करता है। उनके पैमाने (~600M बनाए गए लिंक) पर, उन्होंने 6-अक्षर की-स्पेस का 0.001% से भी कम उपयोग किया है। आपके URL shortener में बढ़ने के लिए काफी जगह है!
</FunFact>

## चरण 4: डेटाबेस स्कीमा

```sql
CREATE TABLE urls (
    id          BIGSERIAL PRIMARY KEY,
    short_code  VARCHAR(7) UNIQUE NOT NULL,
    long_url    TEXT NOT NULL,
    created_at  TIMESTAMP DEFAULT NOW(),
    expires_at  TIMESTAMP,
    user_id     BIGINT,
    click_count BIGINT DEFAULT 0
);

CREATE INDEX idx_short_code ON urls(short_code);
CREATE INDEX idx_expires_at ON urls(expires_at) WHERE expires_at IS NOT NULL;
```

### Sharding रणनीति

100M+ URLs के लिए, एक अकेला डेटाबेस पर्याप्त नहीं है। `short_code` के आधार पर shard करें:

| Shard | सीमा | उदाहरण |
|---|---|---|
| Shard 0 | short_code 0-9 से शुरू | `3xK9mP` |
| Shard 1 | short_code a-m से शुरू | `aB7nQr` |
| Shard 2 | short_code n-z से शुरू | `pL2wXy` |
| Shard 3 | short_code A-Z से शुरू | `Rt5vKm` |

<ThinkAboutIt>
`user_id` के बजाय `short_code` के आधार पर shard क्यों करें? पठन पथ के बारे में सोचें — जब कोई छोटी URL पर क्लिक करता है, तो आपके पास केवल short_code होता है। आपको कुछ और देखे बिना सीधे सही shard पर रूट करना होगा।
</ThinkAboutIt>

## चरण 5: Cache परत

100:1 पठन-से-लेखन अनुपात के साथ, caching महत्वपूर्ण है। यहाँ cache-aside पैटर्न है:

### Cache Hit प्रवाह (सामान्य पथ)
1. अनुरोध आता है: `GET /a1B2c3`
2. Redis जाँचें: `GET url:a1B2c3`
3. Cache HIT → `long_url` लौटाएँ → 301 रीडायरेक्ट
4. कुल विलंबता: **~2ms**

### Cache Miss प्रवाह
1. अनुरोध आता है: `GET /a1B2c3`
2. Redis जाँचें: `GET url:a1B2c3` → MISS
3. डेटाबेस क्वेरी: `SELECT long_url FROM urls WHERE short_code = 'a1B2c3'`
4. Cache में लिखें: `SET url:a1B2c3 "https://..." EX 86400`
5. `long_url` लौटाएँ → 301 रीडायरेक्ट
6. कुल विलंबता: **~15ms**

### Cache कॉन्फ़िगरेशन

```
Cache size:    ~20% of total URLs (hot set)
Eviction:      LRU (Least Recently Used)
TTL:           24 hours (balance freshness vs hit rate)
Expected hit rate: 80-90% (Pareto: 20% of URLs get 80% of traffic)
```

<Callout type="info">
**Cache वार्मिंग रणनीति**: deployment पर शीर्ष 1000 सबसे अधिक क्लिक की गई URLs को cache में पूर्व-लोड करें। यह पुनरारंभ के बाद cache misses की बाढ़ को रोकता है।
</Callout>

## चरण 6: स्केलिंग रणनीतियाँ

### क्षैतिज स्केलिंग चेकलिस्ट

| परत | रणनीति | नोट्स |
|---|---|---|
| **API सर्वर** | LB के पीछे अधिक instances जोड़ें | Stateless — स्वतंत्र रूप से स्केल करें |
| **Database** | Read replicas + sharding | Primary पर लिखें, replicas से पढ़ें |
| **Cache** | Redis Cluster (6+ nodes) | कुंजी वितरण के लिए consistent hashing |
| **ID Generation** | वितरित: range-based या Snowflake | एकल काउंटर बॉटलनेक से बचें |

### लोकप्रिय URLs का प्रबंधन

कुछ URLs वायरल हो जाती हैं — मिनटों में लाखों क्लिक। रणनीतियाँ:

1. **बहु-स्तरीय cache**: L1 (स्थानीय इन-मेमोरी) → L2 (Redis) → DB
2. **Rate limiting**: Backend को अचानक बाढ़ से बचाएँ
3. **CDN**: Edge पर रीडायरेक्ट कैश करें (एनालिटिक्स के साथ सावधान)

## AI का दृष्टिकोण: ML-संचालित URL Shortening

आधुनिक URL shorteners दो प्रमुख क्षेत्रों में AI का उपयोग करते हैं:

### 1. Spam/Phishing पहचान

```python
# Features for ML spam classifier
features = {
    "url_length": len(long_url),
    "has_ip_address": bool(re.match(r'\d+\.\d+\.\d+\.\d+', domain)),
    "subdomain_count": domain.count('.'),
    "uses_https": long_url.startswith('https'),
    "domain_age_days": get_domain_age(domain),
    "similar_to_known_brand": brand_similarity_score(domain),
    "contains_suspicious_keywords": check_keywords(long_url),
}
# Model: Random Forest or BERT-based classifier
# Precision > 99.5% needed to avoid false positives
```

### 2. क्लिक प्रेडिक्शन (Cache Warming)

ML मॉडल भविष्यवाणी करते हैं कि कौन सी URLs ट्रैफिक स्पाइक प्राप्त करेंगी, जिससे पूर्व-रिक्त cache warming संभव होता है:

- **इनपुट features**: दिन का समय, स्रोत प्लेटफॉर्म (Twitter/Reddit), सामग्री श्रेणी, ऐतिहासिक पैटर्न
- **आउटपुट**: अगले घंटे के लिए अनुमानित क्लिक मात्रा
- **कार्रवाई**: अनुमानित वायरल URLs के लिए cache पूर्व-गर्म करें

<FunFact>
Bitly प्रति माह 10 बिलियन+ क्लिक प्रोसेस करता है और दुर्भावनापूर्ण URLs का रीयल-टाइम में पता लगाने के लिए ML का उपयोग करता है। उनका मॉडल छोटी URL बनाए जाने के सेकंडों के भीतर phishing प्रयासों को पकड़ लेता है — पहले शिकार के क्लिक करने से पहले।
</FunFact>

## इंटरव्यू चेकलिस्ट

अपने डिज़ाइन का स्व-मूल्यांकन करने के लिए इसका उपयोग करें:

- [ ] आवश्यकताएँ स्पष्ट कीं (कार्यात्मक + गैर-कार्यात्मक)
- [ ] पैमाने का अनुमान लगाया (URLs/माह, पठन/सेकंड, स्टोरेज)
- [ ] स्वच्छ API डिज़ाइन किया (REST, उचित स्टेटस कोड)
- [ ] तर्क के साथ एन्कोडिंग रणनीति चुनी
- [ ] इंडेक्सिंग के साथ डेटाबेस स्कीमा डिज़ाइन किया
- [ ] स्पष्ट hit/miss प्रवाह के साथ caching जोड़ा
- [ ] Sharding रणनीति पर चर्चा की
- [ ] एकल विफलता बिंदुओं को संबोधित किया
- [ ] निगरानी और अलर्टिंग का उल्लेख किया
- [ ] AI दृष्टिकोण जोड़ा (spam पहचान, प्रेडिक्शन)

<ThinkAboutIt>
यदि URLs को **हटाने योग्य** (GDPR अनुपालन) होना आवश्यक हो तो आप इस डिज़ाइन को कैसे संशोधित करेंगे? Cache invalidation, डेटाबेस में soft deletes बनाम hard deletes, और आप हटाई गई URL के लिए रीडायरेक्ट अनुरोध कैसे संभालेंगे, इसके बारे में सोचें।
</ThinkAboutIt>
